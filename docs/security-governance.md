# Vibe Boot 安全治理规格

## 1. 文档目的

本文定义 Vibe Boot 的安全治理边界，覆盖用户认证、权限、数据权限、密钥管理、AI 调用脱敏、代码生成风险、审计日志、生产模式限制和实现前准入。

Vibe Boot 引入 AI coding 后，安全治理必须前置。AI 可以提高开发效率，但不能绕过权限、安全、审计和人工确认。

## 2. 安全目标

| 目标 | 说明 |
| --- | --- |
| 用户身份可信 | 所有管理端操作必须有登录态 |
| 权限边界清晰 | 菜单、按钮、接口权限一致 |
| 数据访问受控 | 预留部门、本人、全部等数据范围 |
| 密钥不泄漏 | API Key、数据库密码、Redis 密码、TLS 私钥密码不入库明文、不入 Git |
| AI 调用可控 | 发送模型前脱敏，生产包不包含代码编辑或其他开发型 AI 入口 |
| 模型数据最小化 | 只把完成任务必须的数据发送给模型 |
| 变更可审计 | 谁让 AI 做了什么、改了哪些文件、结果如何 |
| 高风险需确认 | 删除数据、改权限、改认证、发布生产必须确认 |

## 3. 威胁模型

| 风险 | 示例 | 治理策略 |
| --- | --- | --- |
| 越权访问 | 普通用户访问管理接口 | 接口权限强校验 |
| 数据越权 | 查看其他部门数据 | 数据权限拦截 |
| 密钥泄漏 | API Key 被提交到 Git | ignored 配置、扫描、脱敏 |
| AI 泄密 | 把敏感业务数据发送给模型 | 上下文脱敏和白名单 |
| AI 误改 | 修改登录、权限、数据库 | 风险分级和人工确认 |
| 供应商风险 | 模型服务不可用或不可信 | 模型网关、可替换、降级 |
| 生产误操作 | 在线改源码或执行脚本 | 生产禁用开发型 AI 能力 |
| 升级失败 | 数据库迁移破坏生产 | 备份、迁移记录、回滚策略 |

## 4. 认证策略

首版认证必须简单稳定。

| 能力 | P0 要求 |
| --- | --- |
| 用户名密码登录 | 必须支持 |
| 登出 | 必须支持 |
| 会话空闲续期/失效 | 必须支持，且不得突破绝对有效期 |
| 密码加密存储 | 必须支持 |
| 登录失败记录 | 必须支持 |
| 登录限流 | P0 必须支持，不依赖验证码才生效 |
| 验证码 | P1，可先预留 |
| 短信/社交登录 | P2，不进入首版 |

首版认证相关决策：

| 决策 | 取舍口径 | 当前结论 |
| --- | --- | --- |
| 权限框架 | Sa-Token 1.45.0 | 已由 ADR-0001 确认 |
| 会话存储 | 生产 Redis，开发可本地内存 | P0 使用服务端会话，不使用 JWT |
| 密码算法 | JDK 17 `PBKDF2WithHmacSHA256` | 不新增 Spring Security Crypto 或其他密码库 |
| 浏览器凭据 | HttpOnly Cookie | 不把会话 Token 放入 localStorage/sessionStorage |
| 跨站防护 | 同源 + Origin 校验 + 会话绑定 CSRF Token | 所有已登录写请求强制执行 |

初始管理员与密码安全：

| 约束 | P0 要求 |
| --- | --- |
| 开发演示账号 | 可使用 `admin`，但必须标注仅限开发 |
| 生产初始密码 | 默认交互输入并二次确认；显式 `-GenerateInitialAdminPassword` 时生成 24 位随机值并仅显示一次；只经 bootstrap 子进程 stdin 传递，不接受文件、环境变量或命令行参数 |
| 首次登录强制改密 | 初始化管理员必须 `password_reset_required=true`，改密前只能访问改密、登出和当前用户基础信息 |
| 密码记录 | 明文密码、随机初始密码和重置密码不得进入 Git、日志、AI 上下文、备份摘要或默认生产包 |
| 密码重置审计 | 记录操作者、目标用户、时间、traceId 和结果，不记录明文密码 |

### 4.1 密码契约

| 项目 | P0 固定值或规则 |
| --- | --- |
| 长度 | 单因素登录密码最少 15 个、最多 128 个 Unicode code point |
| 字符 | 允许可打印 ASCII、空格和 Unicode；按 NFC 规范化后计算和哈希，不裁剪首尾空格 |
| 组合规则 | 不强制大小写、数字、符号混合，允许密码管理器粘贴和自动填充 |
| 弱密码阻断 | 与规范化用户名、产品名 `Vibe Boot` 及其紧凑写法相同或明显派生时拒绝；同时检查项目内置离线常见密码表 |
| 常见密码表 | 项目维护的 UTF-8 资源，P0 至少 1000 条；若采用第三方列表，来源、版本、许可证和 SHA256 必须进入 NOTICE/manifest；P0 不在线查询境外泄漏库 |
| 定期改密 | 不强制定期修改；初始密码、管理员重置、疑似泄漏或安全事件时强制修改 |
| 初始/重置密码 | JDK `SecureRandom` 从 `ABCDEFGHJKLMNPQRSTUVWXYZabcdefghijkmnopqrstuvwxyz23456789!@$%*-_?` 生成 24 位；至少各含一个大写、小写、数字、符号并用同一 CSPRNG 洗牌；只显示一次并设置 `password_reset_required=true` |

密码存储格式固定为：

```text
$pbkdf2-sha256$600000$<base64url-salt>$<base64url-derived-key>
```

| 参数 | P0 要求 |
| --- | --- |
| KDF | `PBKDF2WithHmacSHA256` |
| 迭代次数 | 600000；生产不得低于该值 |
| salt | 每个密码独立使用 `SecureRandom` 生成 16 bytes |
| derived key | 32 bytes |
| 编码 | Base64 URL-safe、无换行；格式中保存算法和迭代次数，支持未来登录后升级 |
| 比较 | 使用常量时间字节比较，不比较编码字符串 |
| 性能 | 在最低支持硬件上单次校验目标低于 1 秒；不得通过降低 600000 次下限解决性能问题 |
| 禁止 | MD5、SHA-1、单次 SHA-256、可逆加密、全局固定 salt、明文或可恢复密码 |

未知用户名也执行一次相同参数的虚拟密码校验，再返回通用错误，降低账号枚举和明显时序差异。P0 不实现用户自助找回密码；由有权限管理员执行受审计的一次性重置。

### 4.2 登录防爆破

| 维度 | P0 默认策略 |
| --- | --- |
| 账号维度 | 同一规范化用户名连续失败 5 次后暂停 15 分钟 |
| IP 维度 | 同一来源 IP 5 分钟最多 20 次登录尝试，超限暂停 10 分钟 |
| 存储 | 生产使用 Redis 原子计数和 TTL；开发可使用进程内实现 |
| 成功登录 | 清除该账号失败计数，不清除来源 IP 的速率窗口 |
| 失败响应 | 用户不存在、密码错误、账号被删除使用同一 `AUTH_0401` 和通用中文提示 |
| 限流响应 | 返回 HTTP 429、`AUTH_0429` 和 `Retry-After`，不得说明账号是否存在 |
| 日志 | 记录规范化用户名摘要、来源 IP、结果、traceId；不记录密码或完整 Cookie |

账号禁用返回通用登录失败信息，后台日志可以记录真实原因。验证码仍为 P1，不能把“尚未实现验证码”当成 P0 不做限流的理由。

### 4.3 会话与 Cookie

| 项目 | P0 契约 |
| --- | --- |
| Token | 每次登录使用 `SecureRandom` 生成至少 32 bytes，Base64URL 编码的不透明随机值；只作为 Redis 会话键，不承载用户信息 |
| JWT | P0 不使用，不引入 Sa-Token JWT 插件，也不需要 Token Secret |
| Cookie | 名称 `VIBEBOOT_SESSION`，Host-only、`Path=/`、HttpOnly、SameSite=Strict；HTTPS 模式必须 Secure |
| 前端存储 | 禁止把 Token 写入 localStorage、sessionStorage、URL、DOM、日志或 IndexedDB |
| 生命周期 | 绝对有效期 8 小时，空闲有效期 30 分钟；活动只续空闲期，不突破绝对有效期 |
| 记住我 | P0 不提供持久化“记住我” |
| 并发会话 | 同一账号最多 3 个有效会话；第 4 次登录使最早会话失效 |
| 登出 | 只失效当前会话并清除 Cookie；返回成功后该 Token 不得再次使用 |
| 全部失效 | 修改本人密码、管理员重置密码、禁用/删除用户必须失效该账号全部会话 |
| 权限变化 | 角色、菜单或用户角色关系变化后立即清理相关权限缓存，不允许旧会话继续使用已撤销权限 |

P0 使用 Redis 服务端会话，因此安装流程不生成“Token Secret”。未来若改为 JWT 或其他签名 Token，必须新增 ADR，并重新设计撤销、轮换、泄漏响应和密钥管理。

### 4.4 CSRF、CORS 与传输

| 项目 | P0 契约 |
| --- | --- |
| 前后端来源 | 生产由后端承载静态资源，保持同源；开发通过 Vite `/api` 代理访问后端 |
| CORS | 默认关闭；P0 不支持任意跨域或 `*` + credentials |
| Origin | 登录以及所有 Cookie 认证的 POST/PUT/PATCH/DELETE 必须携带并精确匹配配置 Origin；缺失、`null`、多值或不匹配都拒绝，P0 不提供绕过名单 |
| CSRF Token | 登录后生成 32 bytes 随机值并绑定服务端会话；`GET /api/auth/csrf` 返回，前端仅保存在内存 |
| CSRF Header | 已登录的 POST/PUT/PATCH/DELETE 必须提交 `X-CSRF-Token`，服务端常量时间比较 |
| 登录接口 | 只接受 `application/json`、合法同源 Origin，并受账号/IP 限流；不依赖 CSRF Token |
| TLS | 生产非回环访问必须 HTTPS；HTTP 只允许开发模式和生产 `local` 模式的本机回环访问 |
| Windows 防火墙 | 默认不由安装器开放；lan 模式仅在管理员显式确认后开放 HTTPS 业务端口，管理端口/数据库/Redis 永不自动开放 |
| 安全响应头 | 至少设置 `X-Content-Type-Options: nosniff`、`Referrer-Policy: no-referrer`、CSP、`Permissions-Policy`；HTTPS 模式设置 HSTS |

生产 CSP 基线固定为：

```text
default-src 'self'; base-uri 'self'; object-src 'none'; frame-ancestors 'none'; form-action 'self'; script-src 'self'; style-src 'self' 'unsafe-inline'; img-src 'self' data: blob:; font-src 'self' data:; connect-src 'self'
```

HTTPS lan 模式返回 `Strict-Transport-Security: max-age=31536000`，P0 不默认添加 `includeSubDomains` 或 preload；local HTTP 模式不得发送 HSTS。`Permissions-Policy` 默认关闭 camera、microphone、geolocation。若后续前端确需新增外部资源或浏览器能力，必须先修改安全文档，不能把 CSP 整体关闭。

若后续确需前后端跨域，必须先修订 ADR 和本文，明确精确 Origin 白名单、credentials、预检缓存、CSRF 和代理信任边界；不得临时放开全局 CORS。

以上密码参数参考 [OWASP Password Storage Cheat Sheet](https://cheatsheetseries.owasp.org/cheatsheets/Password_Storage_Cheat_Sheet.html)，密码长度、限流和会话处理参考 [NIST SP 800-63B](https://pages.nist.gov/800-63-4/sp800-63b.html)。项目不因此宣称通过 NIST、FIPS 或其他合规认证。

## 5. 授权策略

权限模型采用 RBAC。

| 对象 | 说明 |
| --- | --- |
| 用户 | 登录主体 |
| 角色 | 权限集合 |
| 菜单 | 页面入口 |
| 按钮 | 页面操作 |
| 接口权限 | 后端强校验 |
| 数据范围 | 访问数据边界 |

权限约束：

| 约束 | 说明 |
| --- | --- |
| 后端权限是最终边界 | 前端隐藏按钮不是安全 |
| 新接口默认不公开 | 公开接口必须显式标记 |
| 权限标识统一 | `模块:资源:动作` |
| 菜单权限同源 | 代码生成时同时生成前后端权限 |
| AI 不能绕过权限 | AI 生成接口必须加权限 |

## 6. 数据权限策略

首版预留常见数据范围，不必做复杂多租户。

| 数据范围 | 说明 |
| --- | --- |
| 全部数据 | 管理员 |
| 本部门数据 | 当前用户部门 |
| 本部门及下级 | 组织树 |
| 仅本人数据 | 创建人/负责人 |
| 自定义部门 | P1/P2 |

实现约束：

| 约束 | 说明 |
| --- | --- |
| 数据权限表达统一 | 不散落在业务代码 |
| 查询必须可审查 | 不使用不可控字符串拼接 |
| AI 生成查询需标记数据范围 | 生成模块时必须说明数据权限策略 |

P0 最低执行语义：

| 场景 | P0 要求 | 不允许 |
| --- | --- | --- |
| 系统管理查询 | 至少具备数据范围枚举、当前用户部门、部门树和扩展点 | 把数据权限写死在前端 |
| 生成业务模块 | 必须在元模型、交接包和生成摘要中声明数据范围 | 未声明数据范围就生成列表接口 |
| 客户拜访记录演示 | 至少能证明“全部数据、本人数据、本部门数据”的预期差异，未完全接入时必须标记限制 | 用管理员账号替代所有角色验收 |
| 生产业务 AI 数据分析 | 必须遵守当前用户数据范围 | 模型直接读取全库或绕过查询权限 |

数据权限未完全生效时，系统不得把能力标记为“已完成”。任务摘要、演示输出和质量门禁必须写明当前是“已定义但待接入”还是“已验证生效”。

## 7. 密钥管理

| 密钥类型 | 存储策略 |
| --- | --- |
| 模型 API Key | 按 ADR-0002 使用 JDK AES-256-GCM 加密入库；响应只返回 `credentialConfigured` |
| 模型凭据主密钥 | 32-byte 随机值；开发可用环境变量或 ignored local 配置，生产只允许 `config/secrets/model-master.key`，与数据库密文和备份分离 |
| 数据库密码 | `config/secrets/application-prod.secrets.properties`，关闭继承并限制 ACL；运行/迁移账号分离 |
| Redis 密码 | 同一受限 secret 文件，配套 ACL 用户和实例 key 前缀 |
| TLS 私钥密码 | 生产 LAN 模式写入受限 secret 文件，不进入 Git、日志、AI 上下文、manifest、备份或默认生产包 |
| 第三方凭证 | 未单独设计前不得复用模型凭据表；新增类型必须先补安全设计 |

模型凭据加密固定使用 ADR-0002 的 `AES/GCM/NoPadding` 契约。每次加密必须使用新的 12-byte IV 和绑定配置 ID、供应商编码的 AAD；主密钥不得进入数据库、Git、日志、AI 上下文、默认生产包或 API。主密钥缺失或解密失败时，系统只能把 AI 能力标记为不可用并返回 `AI_0503`，不得尝试明文兼容或回退密钥。

禁止事项：

| 禁止 | 说明 |
| --- | --- |
| 禁止提交密钥到 Git | 任意层级 `.env`、真实 local/prod YAML、`install.json`、`config/**/secrets/`、嵌套 TLS/私钥文件和备份目录必须忽略；只提交不含真实值的 `.example`/模板 |
| 禁止日志打印明文密钥 | 密钥完全隐藏；不得以“调试模式”为由显示前后字符 |
| 禁止返回 API Key 明文给前端 | 保存后只返回 `credentialConfigured`，不返回明文、密文或末四位 |
| 禁止 AI 自动读取无关密钥文件 | 上下文读取必须最小化 |

生产运行身份和外部数据服务安全线：

| 项目 | P0 约束 |
| --- | --- |
| Windows 服务 | Procrun 固定 `NT AUTHORITY\LOCAL SERVICE`，目录权限授予 `NT SERVICE\VibeBoot` service SID；服务不得修改 app/runtime/service/scripts/operations |
| MySQL 运行账号 | 只具备目标 schema 的 SELECT/INSERT/UPDATE/DELETE；常驻服务禁用 Flyway |
| MySQL 迁移账号 | 仅由一次性 Jar migrate 子进程使用，只具备目标 schema 必需 DDL/DML，无全局管理权限 |
| Redis ACL | 只允许 `vibe-boot:<instanceId>:*` 和会话/缓存/限流所需命令；禁止 KEYS/FLUSH/CONFIG/MODULE，P0 不调用 Sa-Token `searchData` 全量搜索路径 |
| 非回环连接 | MySQL 固定 VERIFY_IDENTITY；Redis 固定 TLS + CA + ACL，内网不构成明文例外 |
| 恢复与回滚 | 数据库恢复后、服务启动前必须 SCAN + UNLINK 清空本实例 Redis 前缀，使会话、CSRF、限流和权限缓存失效 |
| 备份密钥分离 | 数据库密文可进入 dump，但模型主密钥和全部 secret 永不进入备份；manifest 只记录主密钥指纹 |

## 8. AI 安全策略

### 8.1 上下文脱敏

发送给模型前必须先做数据分类、最小化和脱敏。P0 不做复杂 DLP 系统，但必须防止把整库、整表、完整日志、密钥文件或无关业务数据直接塞进模型上下文。

| 步骤 | P0 要求 |
| --- | --- |
| 数据分类 | 标记为 public/internal/sensitive/secret |
| 最小化 | 只发送当前任务必要字段、摘要或样例，不发送整表和无关附件 |
| 数据权限过滤 | 生产业务 AI 必须先按当前用户数据范围查询，再进入模型上下文 |
| 脱敏 | 对密钥、密码、Token、手机号、身份证和业务敏感字段做替换或掩码 |
| 出境提示 | 使用境外 API Base 或未知供应商时提示数据出境风险 |
| 审计 | 记录发送了哪些上下文类型和脱敏策略，不记录完整敏感请求体 |

数据分类口径：

| 分类 | 示例 | 模型调用策略 |
| --- | --- | --- |
| public | 产品文档、公开接口说明、示例字段 | 可进入开发模式上下文 |
| internal | 业务字段、权限元数据、脱敏后的样例数据 | 可在任务需要时进入上下文 |
| sensitive | 客户姓名、手机号、合同金额、拜访记录、经营数据 | 默认摘要化或脱敏，生产需权限过滤 |
| secret | API Key、密码、Token、私钥、数据库连接串 | 禁止进入模型上下文 |

发送给模型前应处理：

| 类型 | 处理 |
| --- | --- |
| API Key | 替换为 `[REDACTED_API_KEY]` |
| 密码 | 替换为 `[REDACTED_PASSWORD]` |
| Token | 替换为 `[REDACTED_TOKEN]` |
| 手机号/身份证 | P1 脱敏 |
| 业务敏感字段 | 由业务 skill 标记 |

禁止直接发送给模型：

| 禁止内容 | 说明 |
| --- | --- |
| 完整数据库导出 | 包含大量无关或敏感业务数据 |
| 完整操作日志 | 可能包含路径、参数、用户、错误详情 |
| local 配置和密钥文件 | 属于 secret |
| 未经权限过滤的业务列表 | 生产业务 AI 必须遵守当前用户数据范围 |
| 用户上传的未知文件原文 | P0 只允许用户明确选择并提示风险后处理 |

### 8.2 模型供应商治理

| 约束 | 说明 |
| --- | --- |
| 供应商可禁用 | 管理员可关闭某模型 |
| 用量可统计 | token、调用次数、任务 |
| 错误可追踪 | 失败原因记录 |
| 数据出境提醒 | 接入境外模型时提示风险 |
| 供应商数据策略可见 | 模型配置页必须展示供应商、API Base、数据出境风险和用途 |
| API Base SSRF 防护 | 生产只允许解析后全部地址均为公网单播的 HTTPS；开发 HTTP 只允许字面量回环地址；禁止重定向、私网/链路本地/metadata 地址、trust-all 和无界响应 |
| 敏感任务可阻断 | 未确认数据策略时，涉及 sensitive 数据的任务不得调用模型 |
| 生产不打包代码编辑 | 防止通过配置恢复在线改源码入口 |

### 8.3 AI 代码变更治理

| 风险 | 处理 |
| --- | --- |
| 修改权限/登录 | L3，二次确认 |
| 删除数据/字段 | L3，二次确认 + 备份建议 |
| 新增依赖 | L2，说明原因并更新文档 |
| 修改脚本 | L2，说明影响 |
| 普通 CRUD | L1，计划 + 验证 |
| 文档修改 | L0，摘要即可 |

## 10. 文件上传安全

| 约束 | 说明 |
| --- | --- |
| 限制大小 | P0 单文件 20 MB、请求 25 MB，流式读取时再次计数，不能只信 Content-Length |
| 限制容量 | 默认配额 10 GB，至少保留 2 GB 可用磁盘空间，超限停止写入 |
| 三重类型校验 | 扩展名、声明 MIME、magic bytes/UTF-8 内容必须一致 |
| 危险类型拒绝 | 可执行、脚本、HTML、SVG、压缩包、宏文档及非白名单类型全部拒绝 |
| 路径隔离 | 服务端 UUID 生成存储键，规范化路径必须仍位于 storage root 内 |
| Windows 目录安全 | storage root 及子目录拒绝符号链接、junction/reparse point，ACL 只允许服务账号和管理员写入 |
| 文件名处理 | 原名仅作元数据，移除控制字符和路径分隔符，不参与磁盘路径 |
| 下载鉴权 | 只按文件 ID 通过 Controller 下载，不做静态映射和公开 URL |
| 预览隔离 | P0 只预览白名单图片，设置 `X-Content-Type-Options: nosniff` 和私有禁缓存 |
| 删除安全 | DELETING 后立即不可下载；物理删除失败进入 DELETE_FAILED，不伪装成功 |
| 操作审计 | 上传、下载、预览、删除、重试删除记录用户、文件 ID、结果和 traceId，不记录内部绝对路径 |

P0 不包含杀毒扫描或内容净化，因此不能把“扩展名和签名校验通过”描述为文件安全。PDF、文本和 CSV 默认强制下载；只有 jpg/jpeg/png/webp 可内联预览。任何新增文件类型必须先更新 ADR、API 错误码、质量门禁和读者测试。

## 11. 生产模式安全

生产模式可以配置模型供应商，但这只表示允许受控业务 AI 能力，不表示允许开发型 AI。生产环境不得运行外部 AI Coding 工具、平台代码代理、任意 shell 执行器或在线数据库结构修改器。

以下能力不得进入生产包，不能由管理员开关恢复：

| 能力 | 生产状态 |
| --- | --- |
| AI 修改代码 | 不打包 |
| AI 执行脚本 | 不打包 |
| 数据库结构在线修改 | 不打包 |
| 读取服务器任意文件 | 不打包 |
| 远程命令执行 | 不打包 |

生产允许：

| 能力 | 条件 |
| --- | --- |
| 业务 AI 问答 | 管理员启用 |
| 文档问答 | 不含敏感配置 |
| 数据分析 | 数据权限控制 |
| 生成业务文案 | 低风险 |

生产模型能力边界：

| 能力 | 是否允许 | 说明 |
| --- | --- | --- |
| 业务问答、摘要、分类、文案 | 允许 | 必须通过模型网关、权限和脱敏策略 |
| 读取项目源码 | 禁止 | 生产包默认不包含源码，也不向模型暴露源码 |
| 修改源码或生成补丁 | 禁止 | 代码修改必须回到开发模式 |
| 执行 PowerShell/shell | 禁止 | 生产只允许白名单运维脚本由管理员执行 |
| 在线修改数据库结构 | 禁止 | 结构变化只能通过受控升级包和迁移脚本 |

生产 AI 白名单必须按能力暴露，不允许用一个笼统开关恢复开发模式。

| 生产 AI 能力 | 是否允许 | 入口要求 |
| --- | --- | --- |
| 业务问答 | 可选 | 管理员启用，受权限和脱敏控制 |
| 摘要/分类/文案 | 可选 | 只能处理业务数据或用户输入内容 |
| 数据分析 | 可选 | 必须遵守数据权限，不直接暴露数据库结构操作 |
| 模型连接测试 | 允许 | 只测试供应商连通性和配置有效性 |
| 外部 AI 交接包生成 | 禁止 | 交接包只属于开发和实施链路 |
| 代码生成、补丁应用、源码读取 | 禁止 | 必须回开发模式 |
| shell/PowerShell 执行 | 禁止 | 不得由模型触发 |
| 在线 DDL 或任意 SQL 执行 | 禁止 | 只能通过受控升级包和 Flyway 迁移 |

实现时不得把 `model-prod.yml`、模型供应商配置或业务 AI 开关映射为开发型 AI 权限。生产侧即使配置了模型，也只能启用白名单中的业务能力。

生产健康检查同样属于安全边界：

| 入口 | 安全要求 |
| --- | --- |
| `/actuator/**` | 只监听 `127.0.0.1:8081` 管理端口，业务端口不暴露；只开放 health，详情固定关闭 |
| `/api/system/health` | 必须登录并具备 `system:health:info`，只返回稳定状态和脱敏检查项 |
| 健康响应 | 不返回主机名、数据库名、用户名、连接串、Redis Key、服务器绝对路径、配置值、堆栈或原始错误体 |

不得依赖客户端可伪造的 `X-Forwarded-For` 判断回环请求。P0 未定义受信代理链，因此任何“通过代理公开 Actuator”或把管理端口绑定到非回环地址的请求都必须先修订安全文档和 ADR。

## 12. 安全检查清单

| 阶段 | 检查 |
| --- | --- |
| 编码前 | 是否修改安全边界文档 |
| 生成代码前 | 是否涉及权限、认证、数据、密钥 |
| 应用补丁前 | 是否 L2/L3 风险 |
| 构建前 | 是否存在密钥文件 |
| 安装前 | 是否备份数据 |
| 生产运行 | 是否禁用开发型 AI 能力 |
| 生产健康检查 | Actuator 是否仅回环且最小暴露，系统健康接口是否鉴权脱敏 |
| 模型调用前 | 是否完成数据分类、最小化、脱敏和出境提示 |

## 15. 一句话总结

Vibe Boot 的安全治理目标是让 AI coding 成为可审计、可确认、可回滚的工程能力，而不是一个能绕过权限、泄漏密钥或误改生产的黑盒助手。

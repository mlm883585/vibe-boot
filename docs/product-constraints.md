# Vibe Boot 产品约束与实现准入

## 1. 文档目的

本文定义 Vibe Boot 在进入编码实现前必须遵守的产品约束、技术约束、交付约束和实现准入条件。它不是功能清单，而是后续所有设计、编码、重构、发布决策的边界。

只要还没有正式进入某个模块的编码实现，所有新增想法、范围调整、技术选择、部署策略和 AI 工作流变化，都必须优先修订文档，再进入代码实现。

| 约束类别 | 作用 |
| --- | --- |
| 产品约束 | 防止产品做成泛低代码、泛脚手架或泛 AI Chat |
| 技术约束 | 控制技术栈复杂度，保证中小企业可部署、可维护 |
| 平台约束 | 明确首版 Windows 优先，避免过早跨平台分散精力 |
| AI 约束 | 让 AI coding 有边界、有审计、有质量门禁 |
| 交付约束 | 确保开发模式和生产模式都能闭环 |
| 实现准入 | 决定什么时候可以进入签收流程；真正编码仍以签收记录和启动口令为准 |

AI 工具的用户侧使用方式以 `docs/adr/0003-ai-tool-usage-boundary.md` 和 `docs/ai-tool-usage-guide.md` 为准：当前已分层定稿为外部 AI Coding 工具、平台 AI 工作台、模型网关和生产业务 AI；开发模式借助外部 AI Coding 工具，平台内 AI 工作台负责受控业务生成，生产模式禁止开发型 AI 能力。该口径不再作为未定问题悬空，但真正实现仍必须完成人工签收。

## 2. 产品定位约束

Vibe Boot 的定位必须保持清晰：它是面向中国中小企业的 AI 原生 Java 单体应用开发与交付底座。

| 约束 | 说明 |
| --- | --- |
| 不做传统低代码平台 | 不以拖拽、配置解释器、运行时 DSL 作为核心 |
| 不做单纯 Admin 模板 | 权限、菜单、字典、日志只是基础，不是最终差异化 |
| 不做通用 AI Chat | AI 必须围绕项目代码、skills、规则、测试和发布闭环工作 |
| 不做微服务平台 | 首版坚持模块化单体，不引入服务治理复杂度 |
| 不做全行业大而全模板市场 | 首版只做好通用企业管理系统底座 |
| 不做云厂商绑定 | 可接入云能力，但核心运行不依赖某一家云平台 |
| 不做开发者炫技项目 | 首要目标是中小企业用户拿到后低成本跑起来 |

一句话边界：

> Vibe Boot 让用户从一个可运行、可理解、可打包交付的 Java/Vue 单体应用开始，通过 AI coding 持续生成真实代码，而不是被锁在低代码运行时里。

## 3. 目标用户约束

首版不服务所有人，只服务最可能受益且落地阻力最小的用户。

| 用户类型 | 是否首版重点 | 说明 |
| --- | --- | --- |
| 中国中小企业老板/业务负责人 | 是 | 关心低成本、能跑、能改、能交付 |
| 企业内部信息化人员 | 是 | 懂一点业务和电脑操作，但不一定是专业程序员 |
| 独立软件实施人员 | 是 | 需要快速交付客户定制系统 |
| Java 全栈开发者 | 是 | 可接管、扩展和审查 AI 生成代码 |
| 大型企业 IT 架构团队 | 否 | 首版不满足复杂组织治理、集群、高可用要求 |
| 专业低代码平台用户 | 非重点 | 除非其痛点是想摆脱平台锁定 |
| 纯前端/纯 AI Agent 用户 | 非重点 | Vibe Boot 不是独立前端搭建器或 Agent 编排器 |

## 4. 技术栈约束

技术栈必须少而稳。首版每引入一个新技术，都必须回答：它是否降低用户总成本，而不是只降低开发者实现成本。

### 4.1 首版允许技术栈

| 层级 | 允许选择 | 约束 |
| --- | --- | --- |
| JDK | JDK 17 | 固定主版本，不追逐最新 JDK |
| 构建 | Maven 3.8.x | 不同时支持 Gradle |
| 后端框架 | Spring Boot 3.5.16 | P0 锁定 3.5.x 版本线，不自动升级 Spring Boot 4.x |
| ORM | MyBatis-Plus 3.5.16 | 不混用 JPA |
| 数据库 | MySQL 8.x | 首版只支持 MySQL |
| 缓存 | Redis | 仅用于登录态、验证码、限流、轻量缓存 |
| 前端 | Node.js 24.x LTS（基线 24.18.0）+ npm + Vue 3.5.39 + Vite 8.1.3 + TypeScript 6.0.3 + Element Plus 2.14.2 + Pinia 3.0.4 + Vue Router 4.6.4 + Axios 1.18.1 | 不同时支持 React/Angular，不使用 `latest` 或 `*` |
| UI | Element Plus 2.14.2 | 已由 ADR-0001 确认，不允许双 UI 并存 |
| 接口文档 | Springdoc OpenAPI 2.8.17 | 不引入多个接口文档工具 |
| 测试 | JUnit 5、Spring Boot Test；P0 数据库集成测试连接隔离 MySQL 8 测试库 | P0 不用 H2 掩盖方言差异，也不引入 Testcontainers/Docker；Testcontainers 保留 P1 |
| 脚本 | PowerShell | 首版 Windows 优先 |
| Java 样板代码 | P0 不引入 Lombok | 实体、DTO、VO 和配置类使用显式 getter/setter 或 Java 原生能力 |

### 4.2 首版禁止默认引入

| 技术/能力 | 禁止原因 | 未来条件 |
| --- | --- | --- |
| Spring Cloud / Nacos / Sentinel | 增加服务治理和部署复杂度 | 出现明确多服务拆分需求 |
| Spring Boot 4.x | 扩大主版本升级和生态适配风险 | 先更新 ADR、工程骨架、质量门禁和签收包 |
| MQ | 首版业务闭环不依赖异步消息 | 有可靠异步任务和削峰需求 |
| Elasticsearch | 运维成本高 | 出现全文检索或日志检索刚需 |
| Kubernetes | 超出中小企业首版部署能力 | 云端商业版或大客户版本 |
| 多数据库适配 | 测试矩阵膨胀 | MySQL 版本稳定后再扩展 |
| 多前端框架 | 维护成本高 | 生态化阶段再考虑 |
| 多租户复杂模型 | 容易污染所有业务设计 | 明确 SaaS 化后单独设计 |
| 工作流引擎重依赖 | 容易过早复杂化 | 简单状态机无法满足时再引入 |
| Lombok | 增加 IDE 插件、注解处理和生成模板差异 | P1 以后重新评估且必须先更新 ADR |

## 5. Windows 优先约束

首版以 Windows 为第一平台。跨平台能力只能作为架构预留，不能影响 Windows 首版体验。

| 约束 | 要求 |
| --- | --- |
| 开发包必须 Windows 可用 | 用户解压后可通过 PowerShell 启动 |
| 生产包必须 Windows 可安装 | 生产固定 Procrun LocalService + service SID；脚本、package manifest 和文件哈希具备可信签名链 |
| 不要求用户提前安装 JDK/Maven/Node | 开发包内置或脚本自动定位 |
| 默认使用国内镜像 | Maven、npm 必须有国内镜像配置 |
| 默认目录清晰 | `app/`、`runtime/`、`data/`、`logs/`、`scripts/` |
| 不强制 Docker | Docker 可以作为增强方案，不是首版依赖 |
| 不强制管理员权限 | 除安装 Windows 服务外，开发模式应尽量不要求管理员权限 |

Windows 首版体验目标：

| 场景 | 目标体验 |
| --- | --- |
| 第一次启动开发模式 | 解压、执行脚本、配置大模型、进入工作台 |
| 第一次生成业务模块 | 描述需求、确认计划、AI 修改代码、自动验证、本地预览 |
| 第一次生产交付 | 执行构建脚本、复制安装包、执行安装脚本、系统运行 |
| 第一次备份恢复 | 执行备份脚本生成归档，恢复脚本可重建关键数据 |

## 6. 国内用户体验约束

面向中国用户，依赖下载和网络连通性是产品体验的一部分，不是用户自己的问题。

| 项目 | 约束 |
| --- | --- |
| Maven | 默认提供国内镜像 `settings.xml` |
| npm | 默认使用 `npmmirror` registry |
| 依赖锁定 | 前端必须提交锁文件 |
| 后端版本 | Maven 依赖版本必须锁定，不使用动态版本 |
| 模型供应商 | 支持至少一个国内常用大模型供应商 |
| 网络失败 | 脚本必须给出中文错误提示和修复建议 |
| 文档语言 | 首版文档以中文为主 |

## 6.1 开源合规与第三方依赖约束

Vibe Boot 计划以开源仓库和可分发 Windows 包的方式交付，因此第三方依赖、运行时和示例代码必须在编码前建立最低合规边界。这里不做复杂法务系统，但必须让依赖来源、版本、许可证和分发范围可追踪。

| 约束 | 说明 |
| --- | --- |
| 依赖来源可追踪 | Maven、npm、JDK、Node、Procrun 等随包依赖必须记录来源、版本和用途；MySQL/Redis 不随包分发 |
| 许可证摘要可生成 | 开发包和生产包必须能生成或携带第三方依赖许可证摘要 |
| 禁止不明来源二进制 | runtime、脚本工具、exe、zip 不得无来源进入仓库或发行包 |
| 源码仓库不提交大型 runtime | runtime 进入发行包或下载缓存，源码仓库只提交 manifest、说明和脚本 |
| 高风险许可证需阻断 | 发现可能要求闭源业务代码开源、禁止商业分发或来源不明的依赖时，不得继续打包 |
| 第三方源码不直接复制 | 参考开源项目只能吸收设计和模式，不能复制大段源码、资源或文案 |
| 合规清单随包分发 | 开发包必须包含 `runtime/RUNTIME-MANIFEST.json` 与 `runtime/THIRD-PARTY-NOTICES.txt`；生产包必须包含 `notices/RUNTIME-MANIFEST.json`、`notices/DEPENDENCY-MANIFEST.json` 与 `notices/THIRD-PARTY-NOTICES.txt` |

P0 使用上述固定清单、Maven/npm 解析结果和人工许可证复核组合实现；生产 `DEPENDENCY-MANIFEST` 还必须记录交付产物 SHA256。P1 再考虑自动化许可证扫描。许可证结论不构成法律意见，但工程上必须避免“来源说不清、版本说不清、能不能分发说不清”的依赖进入首版。

## 7. 开发模式约束

开发模式是 Vibe Boot 的核心使用场景。它必须让用户在低门槛环境下通过 AI 生成真实代码。

| 约束 | 说明 |
| --- | --- |
| 必须以代码仓库为中心 | AI 的所有修改都进入真实代码文件 |
| 必须先生成变更计划 | 重大修改不能直接写代码 |
| 必须展示 diff 摘要 | 用户能知道 AI 改了什么 |
| 必须运行质量检查 | 至少编译通过，关键路径有测试或明确说明测试缺口 |
| 必须保护密钥 | 模型 API Key、数据库密码不得进入 Git |
| 必须保留回滚点 | 重大修改前建议或自动创建本地检查点 |
| 必须尊重文档优先 | 需求、边界、技术选型不清时先补文档 |
| 必须通过 AI 使用准入 | 编码许可、任务阶段、执行入口、上下文、风险和验证计划必须先确认 |

开发模式不得默认执行：

| 禁止项 | 原因 |
| --- | --- |
| 自动推送远端仓库 | 用户必须确认提交和推送 |
| 自动删除数据库字段或数据 | 数据风险高 |
| 自动修改安全核心逻辑 | 权限、登录、密钥、审计必须人工确认 |
| 自动引入大依赖 | 技术栈约束必须先修订文档 |
| 自动生成不可解释代码 | 用户需要能接管和维护 |
| 跳过 AI 使用准入卡 | 容易把未签收、跨阶段或生产高风险请求误当成普通 AI 对话 |

## 8. 生产模式约束

生产模式必须稳定、克制、可备份、可升级。它不是开发模式的简单启动方式。

| 约束 | 说明 |
| --- | --- |
| 关闭代码编辑入口 | 生产默认不能让 AI 直接修改源码 |
| 使用构建产物运行 | 后端 jar + 前端静态资源 |
| 配置与程序分离 | 生产配置、密钥、数据不得打进 jar |
| 数据可备份 | MySQL、上传文件和非敏感配置可一键备份；全部 secret 与模型主密钥必须排除并独立托管 |
| 升级可回滚 | 目标包签名验证、同卷暂存、原子状态、升级前回滚点和 Redis 会话失效均可判定 |
| 服务最小权限 | 常驻服务只持有 MySQL DML、受限 Redis ACL 和所需目录权限，不可修改自身程序或脚本 |
| 日志可定位 | 关键启动、访问、错误、AI 调用必须有日志 |
| 安装可重复 | 安装脚本重复执行应能检测已有状态并给出提示 |

生产模式 AI 能力约束：

| 能力 | 默认状态 |
| --- | --- |
| 业务问答/RAG | 可选开启 |
| 文案生成/数据分析 | 可选开启 |
| 代码生成 | 不进入生产包，不存在可恢复入口的运行时开关 |
| 数据库结构修改 | 不进入生产业务服务，只能由受控安装/升级中的一次性 migrate 执行 |
| 远程工具执行 | 不进入生产包，不提供 shell、补丁或交接包执行入口 |

开发成果进入生产只能走受控发布通道。

| 发布通道约束 | 要求 |
| --- | --- |
| 唯一入口 | 开发工作区通过质量检查后执行 `build-prod.ps1` 生成生产安装包 |
| 唯一安装方式 | 首装运行已验证目标包的 `install.ps1`；升级只运行已验证目标包的 `upgrade.ps1`，signer thumbprint 来自带外渠道 |
| 唯一结构变更方式 | 数据库结构变化只能由同一 Jar 的一次性 Flyway migrate 维护模式执行；常驻服务无 DDL 权限 |
| 必须可回退 | 安装、升级和高风险迁移前必须有备份或回滚说明 |
| 禁止复制源码上线 | 生产不得直接复制 `backend/`、`frontend/`、开发配置、Maven/Node 环境作为运行方式 |
| 禁止执行交接包上线 | 外部 AI 交接包、补丁、临时 SQL、shell 命令不能作为生产发布入口 |
| 禁止手工改库伪造发布 | 生产结构和基础数据变更必须可追踪、可迁移、可审计 |

## 9. AI 能力约束

AI 是开发协作者，不是无边界自动化执行器。

| 约束 | 要求 |
| --- | --- |
| 上下文必须可追溯 | AI 使用了哪些文档、skills、文件，需要在摘要中说明 |
| 输出必须结构化 | 需求、计划、变更、风险、验证结果分别呈现 |
| 高风险必须确认 | 涉及数据、安全、依赖、发布的修改必须人工确认 |
| 失败必须解释 | 编译失败、测试失败、模型失败都要给中文解释 |
| 模型可替换 | 业务代码不直接绑定某个模型 SDK |
| 成本可统计 | 至少按任务记录模型调用和 token 用量 |
| 敏感信息不外发 | 发送给模型前必须有脱敏策略 |
| 数据最小化 | 不发送整库、完整日志、secret 数据或无关业务数据 |
| 出境风险可见 | 境外或未知模型供应商必须有中文风险提示和确认 |

AI 工具分工约束：

| 工具 | 允许 | 禁止 |
| --- | --- | --- |
| 外部 AI Coding 工具 | 开发模式修改源码、运行验证、解释工程 | 生产服务器在线改代码 |
| 平台内 AI 工作台 | 需求澄清、计划、风险、受控生成、验证摘要 | P0 变成完整 IDE 或任意命令执行器 |
| 模型网关 | 统一模型调用、用量、错误和脱敏 | 业务模块直接散落模型 SDK |
| 生产业务 AI | 摘要、分类、问答、文案、数据分析 | 修改源码、执行脚本、变更数据库结构 |

AI 用户入口约束：

| 用户入口 | 首版定位 | 约束 |
| --- | --- | --- |
| 外部 AI Coding 工具 | 开发模式真实源码修改主路径 | 必须读取文档、输出计划、执行验证、说明风险 |
| 平台内 AI 工作台 | 企业用户和实施人员的产品化入口 | 必须围绕需求、计划、风险、元模型、验证摘要工作 |
| Windows 脚本 | 环境检查、启动、构建、安装、备份恢复入口 | 必须中文提示，不隐藏失败 |
| 生产业务 AI 页面 | 安全业务辅助入口 | 只能通过模型网关调用，不具备开发型能力 |

首版不得把“用户自己会用 AI 工具”当作产品前提。平台必须提供提示词、任务单、阅读顺序、验证命令和失败处理，让外部 AI Coding 工具在可控轨道内工作。

AI 工具使用必须满足以下产品化要求：

| 要求 | 说明 |
| --- | --- |
| 首次使用可引导 | 用户解压、检查环境、配置模型、启动开发模式后，应知道下一步如何让 AI 读取项目上下文 |
| 任务可交接 | 平台 AI 工作台必须能把业务需求转换为外部 AI Coding 工具可执行的交接包 |
| 上下文可复制 | 交接包必须包含阶段、目标、允许范围、禁止事项、风险、验证命令和输出格式 |
| 企业用户不必懂源码 | 企业用户确认业务和风险，源码修改可由实施人员或开发者使用外部 AI 工具完成 |
| 企业用户不必先会 AI Coding 工具 | 平台必须通过工作台托底，生成结构化需求、准入卡、交接包和验收清单 |
| 补丁只限开发工作区 | P0 通用补丁由外部 AI Coding 工具承接；确定性生成器仅写 owned 路径；P1 本地执行器需重新签收，生产始终禁止 |
| 交接包不是生产执行入口 | 外部 AI 交接包只用于开发/实施协作，不能在生产服务器直接执行补丁、SQL 或 shell |
| 交接包不是编码授权书 | 交接包必须同时满足签收状态、阶段启动口令、允许范围和质量门禁，才能进入源码修改 |
| 外部工具不自由发挥 | 外部 AI Coding 工具必须受 `docs/README.md`、ADR、阶段任务和质量门禁约束 |
| 生产不承接开发动作 | 生产模式即使配置模型，也只能启用业务 AI，不能启用代码修改、shell 或在线改表 |

首版 AI 操作模型文档口径已分层定稿，待人工签收后冻结为以下组合；进入编码后不得重新解释成完整 AI IDE、生产 Agent 或普通 Chat 外壳。

| 组成 | 签收后冻结职责 | 不允许滑向 |
| --- | --- | --- |
| 外部 AI Coding 工具 | 在开发模式中执行真实源码修改、命令验证、问题修复和摘要交付 | 平台 P0 自研完整 IDE Agent |
| 平台 AI 工作台 | 把业务需求沉淀为澄清问题、变更计划、风险、元模型、补丁建议和验证摘要 | 只做无项目上下文的 Chat 页面 |
| 模型网关 | 统一模型配置、调用、脱敏、错误、用量和审计 | 业务模块直接写死模型 SDK 或 API Key |
| Skills/规则 | 固化项目约束、生成规范、质量门禁和安全边界 | 让模型自由发挥或靠临时 Prompt 约束 |
| 生产业务 AI | 只服务问答、摘要、分类、文案和分析 | 生产服务器在线改代码、执行脚本或改表结构 |

模型上下文数据约束：

| 数据类型 | 首版处理 |
| --- | --- |
| 项目文档和公开规范 | 可作为开发模式上下文 |
| 业务字段和元模型 | 可在任务需要时发送，优先发送结构化摘要 |
| 客户、合同、拜访、经营数据 | 默认脱敏或摘要化，生产模式必须先经过数据权限过滤 |
| 密钥、密码、Token、连接串 | 禁止进入模型上下文 |
| 完整日志、整库导出、未知上传文件 | P0 默认禁止，除非后续单独设计确认和脱敏流程 |

AI 工具操作模型约束：

| 约束 | 说明 |
| --- | --- |
| 不替代成熟 AI Coding 工具 | 首版不重做 Codex、Cursor、Claude Code、通义灵码的文件编辑、终端和上下文能力 |
| 不要求企业用户直接写 Prompt 工程 | 平台 AI 工作台负责把业务需求转成澄清问题、计划、风险和元模型 |
| 不要求企业用户先学会 Agent 工具 | 用户只需要在平台内表达业务，实施人员或开发者再用外部 AI Coding 工具执行工程动作 |
| 不把工作台做成普通聊天窗口 | 工作台必须围绕项目文档、skills、规则、质量门禁和生成记录工作 |
| 不把外部工具使用变成隐性流程 | 文档必须提供开场提示词、任务边界、验证命令和摘要格式 |
| 不把生产当开发环境 | 生产服务器不运行开发型 AI Coding 工具，不在线修改源码，也不执行外部 AI 交接包 |

AI 工具使用确定性约束：

| 未确定风险 | 当前固定口径 |
| --- | --- |
| 用户不知道该打开哪个 AI 工具 | 企业用户先打开平台 AI 工作台；实施人员或开发者再使用外部 AI Coding 工具执行源码任务 |
| 外部 AI 工具和平台工作台职责重叠 | 工作台负责需求、计划、风险、交接包和摘要；外部工具负责开发工作区内真实源码修改 |
| 企业用户不会 Prompt 工程 | 平台生成结构化交接包，企业用户只确认业务、字段、权限、风险和验收结果 |
| 外部 AI 工具自由发挥 | 交接包必须包含准入卡、允许范围、禁止事项、相关文档、验证命令和输出格式 |
| 生产环境也想继续 vibe coding | 生产只能使用业务 AI；任何代码修改、脚本执行、结构变更都必须回开发模式 |
| 平台 P0 变成完整 Agent IDE | P0 不提供任意文件写入、任意 shell、完整终端或生产执行器 |

AI 使用口径签收约束：

| 签收点 | 约束 |
| --- | --- |
| 文档成文不等于实现许可 | AI 使用模型可以作为签收依据，但未签收前不得实现 AI 工作台、代码生成器或源码目录 |
| 用户不会 AI Coding 不是例外场景 | 平台必须用工作台、交接包和实施人员路径承接，而不是要求企业用户直接操作 IDE Agent |
| 外部工具是执行环境，不是产品边界外的黑盒 | 外部 AI Coding 工具必须读取项目文档、遵守阶段范围、输出验证和风险摘要 |
| 工作台是产品化入口，不是万能执行器 | 工作台负责需求、计划、风险、元模型、交接包和验证摘要；P0 不提供任意文件写入或 shell |
| 生产 AI 与开发 AI 严格分离 | 生产配置模型只代表业务 AI 可用，不代表允许在线改代码、改库或执行交接包 |

AI 修改代码的最低流程：

| 步骤 | 要求 |
| --- | --- |
| 1. 读取上下文 | 读取相关文档、代码、skills |
| 2. 形成计划 | 列出文件、接口、数据库、前端影响 |
| 3. 风险标注 | 标记安全、数据、依赖、兼容风险 |
| 4. 用户确认 | 高风险任务必须等待确认 |
| 5. 应用修改 | 使用补丁或受控生成方式修改代码 |
| 6. 自动验证 | 编译、测试、格式化或说明无法验证原因 |
| 7. 输出摘要 | 变更内容、验证结果、后续建议 |

## 10. 代码生成约束

Vibe Boot 的代码生成不是低代码运行时解释，而是生成可维护的项目代码。

| 约束 | 要求 |
| --- | --- |
| 生成代码必须符合分层 | Controller、Service、Mapper、Entity、DTO/VO 边界清晰 |
| 生成接口必须有权限 | 除公开接口外必须生成权限标识 |
| 生成页面必须可读 | 列表、表单、详情、搜索、操作布局一致，并遵守 `docs/frontend-admin-spec.md` |
| 生成 SQL 必须可迁移 | 不直接让用户手工复制零散 SQL |
| 生成内容必须可重复 | 同一元模型生成结果应稳定 |
| 生成后必须可编辑 | 代码不是模板黑盒，开发者可直接维护 |
| 生成后必须更新文档 | 新增模块至少更新模块说明或变更记录 |
| 生成后必须验证 | 遵守 `docs/quality-gates.md` 的阶段门禁和失败处理 |

## 11. 数据与安全约束

| 领域 | 约束 |
| --- | --- |
| 用户认证 | 首版使用 Sa-Token + Redis 不透明会话、HttpOnly Cookie、空闲续期和可撤销失效；P0 不使用 JWT |
| 密码安全 | 使用 JDK PBKDF2-HMAC-SHA256/600000 次和独立 salt，密码至少 15 个 code point，并做离线弱密码阻断 |
| 登录防护 | 账号/IP 双维度限流、通用失败响应和未知账号虚拟校验为 P0 必做 |
| 浏览器安全 | 生产同源，默认关闭 CORS；写请求校验 Origin 和会话绑定 CSRF Token，Token 不进入 Web Storage |
| 传输安全 | 生产默认仅回环访问；任何非回环 LAN 访问必须启用 HTTPS |
| 权限控制 | RBAC 是基础，接口和菜单权限一致 |
| 数据权限 | 部门/本人/全部等常见范围首版预留 |
| 操作日志 | 新增、修改、删除、导出等关键操作记录 |
| 登录日志 | 成功、失败、IP、时间、原因记录 |
| 密钥管理 | `.env`、本地配置、生产配置均不得提交 Git |
| 数据库变更 | 使用迁移脚本或版本化 SQL |
| 文件上传 | P0 只做本地文件基础服务：20 MB、白名单、三重类型校验、路径隔离、鉴权访问、两阶段删除；不宣称杀毒，不做公开直链或 Office 在线预览 |

## 12. 文档优先约束

从现在开始，到明确进入编码实现前，项目遵守“文档优先”原则。

| 场景 | 处理方式 |
| --- | --- |
| 新增产品能力 | 先更新产品文档、MVP 或模块设计 |
| 调整技术栈 | 先更新技术约束，并说明取舍理由 |
| 修改部署方式 | 先更新 Windows 开发包或发布包设计 |
| 引入新依赖 | 先记录为什么必须引入、替代方案是什么 |
| 发现竞品启示 | 先更新竞品/参考分析，再决定是否吸收 |
| 准备编码 | 先检查实现准入清单是否满足，再完成签收记录并获得精确启动口令 `开始 S1 工程骨架编码` |
| 新增或拆分文档 | 遵守 `docs/documentation-maintenance-guide.md`，同步 README、准入审计和冻结清单 |

文档优先不是不编码，而是避免在边界没想清楚时把复杂性固化进代码。

## 15. 已收敛决策与延后项

以下问题已经给出 MVP 默认口径。进入编码后不得重新打开讨论；如需调整，先更新 ADR、路线图或冻结清单。

| 问题 | 当前口径 |
| --- | --- |
| 前端 UI 使用 Element Plus 还是 Naive UI | 已由 ADR-0001 确认为 Element Plus |
| 权限框架使用 Sa-Token 还是 Spring Security | 已由 ADR-0001 确认为 Sa-Token 1.45.0 |
| 数据库迁移使用 Flyway、Liquibase 还是自研 | 已由 ADR-0001 确认为 Flyway |
| 是否内置 MySQL 安装包 | 不内置；开发和生产均连接用户/实施方提供的 MySQL 8 |
| Windows 服务实现方式 | 已由 ADR-0001 确认为 Apache Commons Daemon Procrun 1.6.1 x64；P0 不与 WinSW/NSSM 并存 |
| Redis Windows 方案 | 开发默认内存降级并可连接外部 Redis；生产固定外部 Redis；发行包和安装器不分发或接管 Redis |
| 代码生成模板风格 | 已由 ADR-0001 确认为 Velocity 2.4.1 |
| AI 工作台嵌入方式 | 首版可作为管理端模块，不单独做桌面客户端 |
| 是否支持多租户 | 首版不做复杂多租户，只预留租户字段/上下文扩展点 |

## 16. 一句话总结

Vibe Boot 的产品约束是：先用最少技术栈、Windows 优先、国内镜像、真实代码、AI 有边界、开发生产双模式，做出一个中国中小企业真正能启动、能修改、能打包、能部署、能备份的 Java 单体应用底座。

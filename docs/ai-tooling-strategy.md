# Vibe Boot AI 工具使用策略

## 1. 文档目的

本文定义 Vibe Boot 如何使用 AI 工具，明确外部 AI coding 工具、平台内 AI 工作台、模型网关、skills/规则和生产环境 AI 能力之间的边界。

这个问题必须在编码前定稿，否则 Vibe Boot 很容易做成普通 Chat 页面、复杂 Agent 平台，或在生产环境引入不可控代码修改能力。

用户侧的具体使用路径见 `docs/ai-tool-usage-guide.md`。本文侧重系统分工，使用指南侧重开发者、企业用户和生产环境分别怎么操作。

## 2. 核心结论

Vibe Boot 首版不自研完整 AI coding runtime。MVP 采用分层策略：

| 层级 | 工具形态 | 首版定位 |
| --- | --- | --- |
| 外部 AI coding 工具 | Codex、Claude Code、Cursor、GitHub Copilot | 开发 Vibe Boot 自身、执行真实源码修改 |
| Vibe Boot AI 工作台 | 管理端内置页面 | 面向用户的受控需求澄清、计划、生成、验证入口 |
| 模型网关 | OpenAI 兼容 API + 国内模型 | 统一模型调用、配置、用量、脱敏、审计 |
| Skills/规则 | 文档 + Markdown skills + 规则集 | 约束 AI 行为和项目边界 |
| 生产 AI 能力 | 业务问答、文案、分析 | 默认不允许代码修改和脚本执行 |

一句话：

> P0 先让 Vibe Boot 成为外部 AI coding 工具的最佳 Java Admin 工作区，同时在平台内提供一个轻量、受控、可审计的 AI 工作台。

## 2.0.1 当前决策状态

“如何使用 AI 工具”在编码前文档阶段按本文和 `docs/ai-tool-usage-guide.md` 固化为以下结论。它仍需要人工签收才能进入实现，但不再作为未定问题悬空。

| 问题 | 当前结论 | 后续动作 |
| --- | --- | --- |
| 用户到底用什么 AI 工具开发 | 开发者和实施人员使用外部 AI Coding 工具，平台提供上下文、规则和交接包 | 签收后按阶段实现交接包和工作台能力 |
| 企业用户不会 Codex/Cursor 怎么办 | 企业用户使用平台 AI 工作台表达业务、确认计划和验收结果 | S4 实现工作台需求澄清、计划和交接包 |
| 平台是否自己做完整 AI IDE | P0 不做 | 如需改变，先重开 ADR-0003 |
| AI 是否能在生产继续改代码 | 不能 | 生产只保留业务 AI 白名单能力 |
| 外部 AI 交接包是否等于授权 | 不等于 | 必须同时满足签收、阶段启动、范围和质量门禁 |

## 2.1 产品化边界

“使用 AI 工具”在 Vibe Boot 中不是一句建议，而是一条被产品化的开发链路。

| 环节 | 产品化方式 | 不产品化的部分 |
| --- | --- | --- |
| 任务输入 | 工作台提供需求模板、澄清问题、风险提示 | 不要求企业用户写复杂 Prompt |
| 上下文组织 | `docs/README.md`、ADR、任务分解、skills 形成默认上下文 | 不依赖用户临时整理一堆文件 |
| 真实改代码 | P0 交给外部 AI Coding 工具或本地受控脚本 | 不在服务端开放任意文件系统和终端 |
| 变更确认 | 工作台记录计划、文件清单、风险等级和确认动作 | 不允许模型自行决定高风险操作 |
| 验证闭环 | 固定 Maven、npm、PowerShell 验证命令和摘要格式 | 不用“看起来能运行”代替验证 |
| 生产交付 | 构建生产包、安装、备份、恢复、升级 | 不在生产服务器上 vibe coding |

首版体验目标不是“平台自己像 IDE 一样无所不能”，而是让现成 AI Coding 工具在 Vibe Boot 的文档、规则、脚本和工作台约束下稳定产出真实可交付代码。

## 2.2 AI 工具使用的产品假设

Vibe Boot 不能把“用户天然会用 AI Coding 工具”当作首版前提。平台必须把 AI 使用方法产品化到文档、工作台、任务单和脚本里。

| 假设 | 产品约束 |
| --- | --- |
| 企业用户多数不会直接操作 IDE Agent | 工作台必须承接需求输入、澄清、计划和风险确认 |
| 实施人员可能会使用外部 AI Coding 工具 | 工作台必须能输出可复制的外部 AI 交接包 |
| Java 开发者可以接管复杂问题 | 生成代码必须是真实 Java/Vue/SQL 文件，不是低代码黑盒 |
| AI 工具能力差异很大 | 文档必须定义统一阅读顺序、任务边界、验证命令和输出格式 |
| 模型供应商会变化 | 所有平台内调用必须通过模型网关，不把 SDK 散落业务代码 |
| 生产环境更重稳定性 | 生产模型配置只代表业务 AI 能力，不代表开发型 AI 能力 |

因此，首版 AI 使用链路按“企业用户确认业务、平台组织上下文、外部 AI 执行源码、平台记录结果”的方式设计。后续即使引入受控 Patch Manager 或 Verification Runner，也只能运行在开发工作区，不得改变生产禁用开发型 AI 的底线，也不得扩展为平台服务端任意文件写入或任意 shell。

## 3. 为什么不首版自研完整 Agent

| 原因 | 说明 |
| --- | --- |
| 实现复杂 | 文件系统、Git、Patch、终端、测试、权限隔离都很复杂 |
| 安全风险 | 自研 Agent 如果边界不清，容易误删数据或执行危险命令 |
| 已有工具成熟 | Codex、Claude Code、Cursor 已具备代码修改和上下文能力 |
| MVP 目标不同 | 首版重点是 Java Admin + Windows 交付闭环，不是重造 IDE Agent |
| 用户价值更快 | 先用外部 AI 工具开发底座，再把高频流程产品化到平台内 |

## 4. 三种使用场景

### 4.1 开发 Vibe Boot 自身

| 项目 | 策略 |
| --- | --- |
| 使用者 | 项目开发者 |
| 工具 | Codex、Claude Code、Cursor 等 |
| 上下文 | `docs/`、源码、脚本、测试 |
| 能力 | 修改源码、运行命令、补测试、提交代码 |
| 约束 | 必须遵守文档优先、ADR 和编码准入 |

这是当前阶段的主要使用方式。

### 4.2 企业用户生成业务模块

| 项目 | 策略 |
| --- | --- |
| 使用者 | 企业用户、实施人员 |
| 工具 | Vibe Boot AI 工作台 |
| 输入 | 自然语言业务需求 |
| 输出 | 需求澄清、变更计划、风险、生成结果、验证摘要 |
| P0 执行方式 | 平台生成元模型和计划，代码修改可由本地受控执行器/外部 Agent 完成 |
| P1 执行方式 | 开发工作区内置受控 Patch Manager，不提供生产在线改源码 |

### 4.3 生产环境使用 AI

| 项目 | 策略 |
| --- | --- |
| 使用者 | 企业管理员、业务用户 |
| 工具 | 业务 AI 功能 |
| 允许 | 业务问答、文案生成、数据分析 |
| 禁止 | 修改源码、执行脚本、改数据库结构 |
| 默认 | `vibe.ai.dev-tools-enabled=false` |

## 5. P0 AI 工作台能力边界

P0 工作台只做必要闭环，不做完整 IDE。

| 能力 | P0 |
| --- | --- |
| 配置模型 | 支持 |
| 测试模型连接 | 支持 |
| 输入需求 | 支持 |
| 需求澄清 | 支持 |
| 生成变更计划 | 支持 |
| 风险分级 | 支持 |
| 生成元模型 | 支持 |
| 生成代码补丁 | 支持，但可交给本地执行器/外部 Agent 应用 |
| 展示 diff | 支持基础文本 diff |
| 执行终端命令 | 不在平台服务端直接执行，交给本地开发模式脚本/Agent |
| 生产环境代码修改 | 禁止 |

## 6. P1 AI 工作台增强

| 能力 | 说明 |
| --- | --- |
| Patch Manager | 开发工作区内受控应用补丁 |
| Verification Runner | 开发工作区内执行编译、测试、构建 |
| Diff Viewer | 更友好的差异展示 |
| Task Replay | 历史任务复现 |
| Context Index | 文档/代码索引 |
| AI 生成测试 | 自动补测试 |

P1 前仍不允许生产环境执行代码修改。

## 7. 工具分工

| 工作 | 外部 AI coding 工具 | Vibe Boot AI 工作台 | 模型网关 |
| --- | --- | --- | --- |
| 修改 Vibe Boot 源码 | 主负责 | 不负责 | 不负责 |
| 读取 docs 约束 | 主负责 | 主负责 | 不负责 |
| 生成业务模块需求计划 | 可做 | 主负责 | 提供模型 |
| 应用代码补丁 | P0 主负责 | P1 增强 | 不负责 |
| 运行 Maven/npm | 主负责 | P1 可调度 | 不负责 |
| 模型配置和用量统计 | 不负责 | 展示 | 主负责 |
| 生产业务问答 | 不负责 | 主负责 | 主负责 |

## 8. 用户体验路径

### 8.1 当前开发者路径

当前未签收时，本路径只用于文档修订、上下文准备和开工前审计。只有 `docs/coding-start-signoff.md` 已签收且维护者发出启动口令后，外部 AI Coding 工具才可进入源码实现。

| 步骤 | 操作 |
| --- | --- |
| 1 | 打开 Codex/Cursor/Claude Code |
| 2 | 让 AI 读取 `docs/README.md` |
| 3 | 签收前只修订文档；签收后按 ADR 和规格实现 S1 工程骨架 |
| 4 | 运行文档规定的验证命令 |
| 5 | 变更后更新文档和代码 |

### 8.2 MVP 用户路径

| 步骤 | 操作 |
| --- | --- |
| 1 | 解压 Windows 开发包 |
| 2 | 执行 `scripts/doctor.ps1` |
| 3 | 执行 `scripts/dev-start.ps1` |
| 4 | 打开管理端 AI 工作台 |
| 5 | 配置模型 API Base、API Key、模型名 |
| 6 | 输入“帮我做客户拜访记录模块” |
| 7 | AI 澄清需求并生成计划 |
| 8 | 用户确认 |
| 9 | 生成代码或生成补丁交给本地执行器 |
| 10 | 执行验证并展示结果 |

### 8.3 生产用户路径

| 步骤 | 操作 |
| --- | --- |
| 1 | 管理员安装生产包 |
| 2 | 默认不启用开发型 AI 工具 |
| 3 | 如需业务 AI，配置模型 |
| 4 | 使用业务问答/文案/分析 |
| 5 | 代码修改必须回到开发模式 |

## 9. 模型接入策略

| 项目 | 策略 |
| --- | --- |
| 协议 | OpenAI 兼容接口优先 |
| 供应商 | 支持国内模型和国际模型 |
| 配置位置 | 管理端模型配置 + local/prod 配置 |
| 密钥 | P0 ignored local 配置，P1 加密存储 |
| 用量 | 按任务记录模型、token、耗时、成功失败 |
| 生产 | 可禁用模型或禁用开发型能力 |

## 10. 上下文策略

AI 工具必须优先读取：

| 上下文 | 说明 |
| --- | --- |
| `docs/README.md` | 文档索引和编码闸门 |
| ADR | 技术决策和实现契约 |
| 产品约束 | 做什么、不做什么 |
| 模块设计 | 目录和依赖边界 |
| API/数据库规范 | 生成代码统一性 |
| 演示规格 | 首个端到端用例 |

外部 AI coding 工具和平台内 AI 工作台都必须遵守同一套文档上下文。

## 11. 安全边界

| 场景 | 规则 |
| --- | --- |
| 开发模式 | 可修改代码，但必须 diff、验证、摘要 |
| 生产模式 | 默认禁止代码修改、脚本执行、数据库结构修改 |
| 密钥 | 不进入 Git，不进入模型上下文 |
| 高风险操作 | L2/L3 必须确认 |
| 外部工具 | 由开发者负责确认命令和文件变更 |
| 平台内工作台 | 必须记录任务、上下文、模型、风险、验证结果 |

## 12. 不做事项

| 不做 | 原因 |
| --- | --- |
| 不首版自研 IDE | 复杂且偏离 MVP |
| 不在生产跑 coding agent | 安全风险高 |
| 不让业务用户直接执行 shell | 风险不可控 |
| 不绕过 Git/diff | 变更必须可审查 |
| 不把模型 SDK 散落业务模块 | 必须经过模型网关 |

## 14. 编码准入

进入 AI 工具相关编码前必须确认：

| 条件 | 状态 |
| --- | --- |
| 外部 AI coding 工具和平台内工作台分工 | 文档口径已由本文确认，仍需签收记录确认 |
| P0 不自研完整 Agent | 已由本文确认 |
| 生产禁用代码修改 | 已由本文和安全治理确认 |
| 模型网关统一接入 | 已由本文和 AI 工作台设计确认 |
| 外部工具读取文档入口 | 已由本文确认为 `docs/README.md` |

## 15. 一句话总结

Vibe Boot 首版使用 AI 的正确方式是：外部 AI coding 工具负责开发实现和真实源码修改，Vibe Boot 内置 AI 工作台负责把这种能力产品化为受控、可审计、可验证的业务模块生成流程。

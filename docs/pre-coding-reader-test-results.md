### 2.1 两轮审阅记录

第一轮独立审阅结论为 **FAIL**。以下阻塞项都必须保留在审计记录中，不能因为后续已修订而改写为“第一轮通过”。

| 编号 | 第一轮阻塞项 | 修订后的权威证据 | 第二轮结果 |
| --- | --- | --- | --- |
| B01 | 首次执行 `install.ps1` 前缺少包外可信根，包内哈希存在自证循环 | `release-package-design.md` 5.1 节固定 OS-only Authenticode 引导和带外 signer thumbprint | 通过 |
| B02 | 升级仅有粗粒度 phase，无法证明多资源提升中断后的原子恢复 | `release-package-design.md` 12 节固定 state v2、九资源子状态、maintenance.flag 和崩溃恢复矩阵 | 通过 |
| B03 | `operations/` 若位于服务可写的 `data/` 下，升级状态和可信启动器可被服务替换 | `release-package-design.md` 7 节固定兄弟目录、关闭继承、显式 ACE 与负向权限测试 | 通过 |

| B09 | P0 API、逻辑 DDL 和生成元模型仍要求编码者补设计 | API/后台/模型/工作台/演示规格、`database-baseline.md` 和 `contracts/codegen-meta-model-v1.schema.json` 已冻结到字段级 | 通过 |

| S2-S7 如何获得阶段授权 | 上一阶段关闭证据通过，`stageAdmission` 的基线/范围/授权人/时间完整，并收到 `post-coding-change-control.md` 冻结的逐字口令；近义表达和任务状态都不算授权 | 通过 |

| P0 模型 API Key 如何保存 | 请求字段 `apiKey` 只写；使用 JDK AES-256-GCM、随机 12-byte IV、128-bit tag 和冻结 AAD 加密为 `credentialCiphertext`，主密钥通过环境变量或外置配置提供；API 只返回 `credentialConfigured` | 通过 |
| 模型主密钥缺失或错误怎么办 | 返回 `AI_0503`，阻断模型保存、连接测试和调用；不得回退到明文、默认密钥或只保存末四位 | 通过 |
| P0 默认命名和端口是什么 | 产品名 `Vibe Boot`、工程标识/后端应用名 `vibe-boot`、Windows 服务名 `VibeBoot`、默认生产安装目录 `C:\VibeBoot`、数据库名 `vibe_boot`、业务 8080、Actuator 管理 8081、前端开发 5173、MySQL 3306、Redis 6379；生产密码、TLS 私钥密码和模型 API Key 不得有公开默认值 | 通过 |
| P0 密码如何保存 | JDK 17 PBKDF2-HMAC-SHA256，600000 次迭代、每密码独立 16-byte salt、32-byte derived key；禁止快速摘要、可逆加密和明文 | 通过 |
| 浏览器会话保存在哪里 | 随机不透明 Token 只在 Redis 和 HttpOnly `VIBEBOOT_SESSION` Cookie 中；响应体和 Web Storage 不保存 Token，P0 不使用 JWT/Token Secret | 通过 |
| P0 如何阻止登录爆破 | 账号连续失败 5 次暂停 15 分钟，同一 IP 5 分钟最多 20 次并暂停 10 分钟；返回 `AUTH_0429` 和 `Retry-After`，未知账号不泄漏存在性 | 通过 |
| 为什么写请求还需要 CSRF Token | 浏览器使用 Cookie 会自动携带凭据，因此已登录写请求必须同时校验同源 Origin 和会话绑定 `X-CSRF-Token` | 通过 |
| 生产能否直接开放 LAN HTTP | 不能。默认 local 模式只绑定 127.0.0.1；lan 模式必须启用 HTTPS 并配置 PKCS12、匹配主机名和精确 allowedOrigin | 通过 |
| lan 安装是否会自动开放所有相关端口 | 不会。防火墙默认不变；只有管理员显式确认时开放产品 HTTPS 业务端口，绝不自动开放 8081、MySQL 或 Redis | 通过 |
| 首版是否允许生产在线改代码 | 否 | 通过 |
| 生产是否允许使用 `admin/admin` 或公开固定初始密码 | 否。开发模式可有 `admin` 演示账号；生产默认交互输入并二次确认，只有显式 `-GenerateInitialAdminPassword` 才生成一次性 24 位值并仅显示一次，随后只经 stdin 传给 bootstrap-admin；首次登录强制改密 | 通过 |
| P0 如何处理备份、跨版本恢复和升级迁移失败 | 备份在停服后创建并按敏感运维资产保护；日常恢复只支持同一完整产品版本；迁移启动后的升级失败必须用同一 rollbackPoint 整套恢复九类资源、SCM、数据库、文件和非敏感配置并清空本实例 Redis，不能只换 jar | 通过 |
| 模型调用企业业务数据前必须做什么 | 数据分类、上下文最小化、脱敏、当前用户数据权限过滤和出境风险提示；secret 数据禁止进入模型上下文 | 通过 |
| 第三方依赖和 runtime 进入开发包或生产包前必须具备什么 | 开发包有 runtime manifest/NOTICE；生产包有 runtime manifest、依赖 manifest、NOTICE；来源、版本、许可证和 SHA256 可追踪 | 通过 |
| 生产包如何建立可信来源，而不是只相信包内哈希 | 管理员从带外渠道取得 signer thumbprint；生产脚本和 `PACKAGE-MANIFEST.psd1` 必须 Authenticode 有效且 signer 一致，签名 manifest 再覆盖包内文件 SHA256；未签名测试包不得生产安装 | 通过 |
| 首次安装时，在 `install.ps1` 尚未执行前如何建立信任 | 管理员在提升权限的 Windows PowerShell 5.1 中使用 OS-only 引导，通过 Windows Authenticode API 校验入口为 Valid，并逐字节比对带外 signer thumbprint；包内哈希和包内 signer 声明不能成为信任根 | 通过 |
| PowerShell 5.1 如何读取并校验脚本配置 | `ConvertFrom-Json` 只做可读性初筛；已认证的同一 Java classpath 必须从文件重新读取原始 JSON bytes，以 strict duplicate detection 和 `docs/contracts/install-v1.schema.json` 做权威校验；Spring Boot 才读取 YAML | 通过 |
| `install.json` 的字段和目录能否由实现者临场补充 | 不能；唯一结构契约是 `docs/contracts/install-v1.schema.json`，密码不在 schema，storage/backup/operations/staging/certs/secrets 从 installRoot 派生，未知字段和非法组合必须拒绝 | 通过 |
| Windows 服务以什么身份运行，谁能修改程序和数据 | Procrun 固定 LocalService 并启用 `NT SERVICE\VibeBoot` service SID；安装根和各子目录关闭继承并写显式 ACE；服务对程序只读、仅对 data/logs 可写，operations 是 data 的兄弟目录且仅管理员可访问 | 通过 |
| 生产 MySQL 为什么需要两个账号 | 常驻运行账号只有 DML；迁移账号只注入一次性 Jar migrate 子进程并具备目标 schema 必需 DDL/DML，常驻服务禁用 Flyway | 通过 |
| 备份是否包含模型主密钥和基础设施密码 | 否；数据库 dump 可含模型密文，但 `config/secrets/**`、模型主密钥、密码和私钥全部排除，manifest 只记录主密钥指纹 | 通过 |
| 数据库恢复后 Redis 怎么处理 | 服务启动前调用同一应用 classpath 的 `--vibe.operation=clear-redis-namespace`，只对派生前缀执行 SCAN、逐 key 复核和分批 UNLINK；禁止 PowerShell 自拼协议、redis-cli、KEYS、FLUSHDB/FLUSHALL，失败保持停服 | 通过 |
| 初始管理员由谁创建、密码如何传递 | Flyway 不写密码；migrate 后同一 Jar 的 `bootstrap-admin` 从 stdin 接收初始密码，事务创建 admin 并设置强制改密 | 通过 |
| Vibe Boot 是否替代 Codex/Cursor/Claude Code | 否，P0 不自研完整 AI IDE，外部 AI Coding 工具是开发模式真实源码修改主路径 | 通过 |
| 企业用户不懂源码时如何使用 AI | 通过平台 AI 工作台描述需求、回答澄清、确认计划和查看结果；源码修改由实施人员或外部 AI Coding 工具在开发模式处理 | 通过 |
| AI 工具责任边界是什么 | 企业用户走平台 AI 工作台，实施人员或开发者使用外部 AI Coding 工具，平台负责交接包和验证摘要，生产用户只使用业务 AI | 通过 |
| AI 工具使用方式是否还没有确定 | 文档口径已成文，但人工签收仍未完成；未签收前不得据此编码 | 通过 |
| 平台是否可以假设用户自己会组织 AI 工具上下文 | 否，平台必须提供首次使用引导、阅读顺序、任务单、验证命令和失败处理 | 通过 |
| 工作台交给外部 AI Coding 工具执行时必须提供什么 | 外部 AI 交接包，包括任务阶段、业务目标、澄清结论、允许范围、禁止事项、风险等级、验证命令和输出格式 | 通过 |
| 每次把任务交给 AI 前必须先确认什么 | AI 使用准入卡：编码许可、任务阶段、执行入口、上下文、风险、验证和生产边界 | 通过 |

| 前端重复提交两次相同角色编码的创建请求 | 数据库唯一索引只允许一条记录，冲突统一映射 `DATA_0409`；不得依赖“先查后插”作为唯一防线 | 通过 |
| 客户端传入自己伪造的 traceId | 忽略客户端值，由服务端生成并返回 `X-Trace-Id`；不得让用户输入污染 MDC | 通过 |
| 用户要求 P1 顺便生成 Excel 导入导出 | 拒绝直接提前，通用导入导出已经归入 P2；如要提升优先级先修订路线图、生成设计和签收材料 | 通过 |
| 用户说“AI 工具我不会配上下文，你自己看着办” | 不把责任推给用户；按首次使用路径引导读取 README、配置模型、选择任务并生成交接包 | 通过 |
| 外部 AI 工具只输出“已完成”但没有验证 | 不通过，要求补充文件清单、验证命令、验证结果和风险摘要 | 通过 |
| S1 编码中用户说“顺便加个消息通知依赖” | 标记为 C2/C3，暂停编码；先修订 ADR、模块设计、路线图和冻结清单 | 通过 |
| 验证命令失败 | 不宣称完成，输出失败摘要和下一步 | 通过 |

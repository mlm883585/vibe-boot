# Vibe Boot S2 基础后台任务分解

## 1. 文档目的

本文把 `docs/basic-admin-spec.md` 拆解为可执行的 S2 编码任务，明确任务顺序、依赖关系、交付物、验证方式和禁止越界项。

S2 的目标不是做一个功能最多的 Admin，而是建立后续 AI 工作台、模型网关、代码生成和业务模块生成都必须依赖的企业应用基础盘。

## 2. S2 总目标

| 目标 | 验收口径 |
| --- | --- |
| 管理员可登录 | 用户名密码登录、登出、当前用户信息可用 |
| 权限可闭环 | 用户、角色、菜单、按钮、接口权限一致 |
| 菜单可驱动前端 | 登录后由后端菜单生成前端路由和按钮权限 |
| 基础数据可维护 | 部门、字典、参数可管理 |
| 行为可审计 | 登录日志、操作日志可查询 |
| 文件基础可用 | 本地单文件上传、鉴权下载、图片预览、配额与两阶段删除可用 |
| 支撑后续 AI 生成 | 菜单、权限、字典、操作人、数据权限扩展点可复用 |

## 3. 前置条件

S2 不应绕过 S1。进入 S2 前必须先满足：

| 前置项 | 要求 |
| --- | --- |
| S1 工程骨架 | 后端 Maven 多模块、前端 Vue 工程和脚本骨架已存在 |
| 阶段签收与启动 | S1 阶段关闭证据已通过，`stageAdmission` 记录完整，并收到精确口令 `开始 S2 基础后台编码`；本文本身不是当前编码许可 |
| 后端构建 | 后端快速构建可执行 |
| 前端构建 | `npm run build` 可执行 |
| 数据库连接 | MySQL 8 开发库可连接 |
| Redis 连接 | Sa-Token 会话所需 Redis 或开发替代方案可用 |
| 文档确认 | `basic-admin-spec.md`、`api-conventions.md`、`database-baseline.md` 已确认 |

| 顺序 | 任务 | 主要产物 | 依赖 |
| --- | --- | --- | --- |

| 2 | 通用安全基础 | 当前用户上下文、权限注解/封装、密码 hash、Token 配置 | 系统表 |
| 3 | 认证接口 | 登录、登出、当前用户、菜单、权限接口 | 安全基础 |
| 4 | 前端登录与布局 | 登录页、管理端布局、动态菜单、权限 Store | 认证接口 |
| 5 | 用户与角色管理 | 用户 CRUD、角色 CRUD、分配角色、分配菜单 | 认证与菜单 |
| 6 | 菜单与部门管理 | 菜单树、按钮权限、部门树 | 权限模型 |
| 7 | 字典与参数配置 | 字典类型、字典项、参数配置 | 基础 CRUD 模式 |
| 8 | 登录日志与操作日志 | 日志记录、日志查询、敏感信息脱敏 | 认证与操作拦截 |
| 9 | 数据权限基础 | 数据范围枚举、当前用户部门上下文、查询扩展点 | 部门与角色 |

| 11 | S2 验收与修订 | 后端/前端验证、问题修复、文档同步 | 全部 S2 功能 |

## 5. 任务明细

### 5.1 系统表与初始数据

| 项目 | 要求 |
| --- | --- |
| 数据表 | `sys_user`、`sys_role`、`sys_menu`、`sys_user_role`、`sys_role_menu`、`sys_dept`、`sys_dict_type`、`sys_dict_item`、`sys_config`、`sys_login_log`、`sys_oper_log`、`file_object` |
| 迁移方式 | Flyway 版本化 SQL |
| 初始数据 | 管理员角色、默认部门、系统管理菜单、文件管理菜单与权限、基础字典 |
| 密码策略 | 开发模式可使用 `admin` 演示账号；生产默认交互输入并二次确认，只有显式 `-GenerateInitialAdminPassword` 才生成 24 位随机值并仅显示一次，随后只经 bootstrap-admin stdin 传递 |
| 首次登录 | 生产初始化管理员必须 `password_reset_required=true`，首次登录后强制修改密码 |
| 索引 | 用户名、角色编码、菜单权限标识、字典编码和关系表组合键等唯一索引 |
| 并发字段 | 可编辑主表包含 version，默认 0；更新 SQL 匹配旧 version 并原子加一 |

交付物：

| 类型 | 内容 |
| --- | --- |
| SQL | 系统表迁移脚本、初始数据脚本 |
| 后端 | Entity、Mapper 基础结构 |
| 文档 | 如字段与 `database-baseline.md` 冲突，先修订文档 |

### 5.2 通用安全基础

| 项目 | 要求 |
| --- | --- |
| 权限框架 | 使用 Sa-Token，业务代码通过 `vibe-security` 封装访问 |
| 会话 | Sa-Token + Redis 服务端不透明随机 Token；通过 HttpOnly Cookie 传递，不返回到响应体或 Web Storage |
| 密码 | JDK PBKDF2-HMAC-SHA256/600000 次、独立 16-byte salt、32-byte derived key，不保存明文或可逆密文 |
| 登录限流 | 账号连续失败和来源 IP 双维度 Redis 原子计数，超限返回 `AUTH_0429` + `Retry-After` |
| CSRF | 会话绑定随机 Token；所有已登录写请求校验 Origin 和 `X-CSRF-Token` |
| 当前用户 | 提供 userId、username、deptId、roleCodes、permissions |
| 权限校验 | Controller 使用统一注解或封装，不散落原始框架调用 |
| 无权限响应 | 返回 `AUTH_0403` |

### 5.3 认证接口

| API | 要求 |
| --- | --- |
| `POST /api/auth/login` | 用户名密码登录，成功记录日志 |
| `POST /api/auth/logout` | 登出并使 Token 失效 |
| `POST /api/auth/change-password` | 修改本人密码并使该账号全部会话失效 |
| `GET /api/auth/csrf` | 返回当前会话绑定的 CSRF Token，前端仅保存在内存 |
| `GET /api/auth/me` | 当前用户、角色、部门、基础信息 |
| `GET /api/auth/menus` | 当前用户可见菜单树 |
| `GET /api/auth/permissions` | 当前用户按钮/接口权限标识 |

验收点：

| 场景 | 结果 |
| --- | --- |
| 正确账号密码 | 登录成功，设置 HttpOnly `VIBEBOOT_SESSION` Cookie，响应体不返回 Token |
| 错误密码 | 中文错误，写入登录失败日志 |
| 不存在账号 | 与错误密码使用同一 `AUTH_0401` 和近似校验耗时，不泄漏账号存在性 |
| 超出登录限制 | 返回 HTTP 429、`AUTH_0429` 和 `Retry-After` |
| 禁用用户 | 不允许登录 |
| 未登录访问 | 返回 `AUTH_0401` |
| 初始化密码登录 | 只允许访问改密、登出和当前用户基础信息，改密完成后才能进入管理端 |
| 缺少 CSRF | 已登录写请求返回 `AUTH_0403`，不得执行业务操作 |

### 5.4 前端登录与布局

| 页面/能力 | 要求 |
| --- | --- |
| 登录页 | 中文表单校验，登录失败展示后端 message |
| 管理端布局 | 左侧菜单、顶部用户菜单、主内容区 |
| 动态路由 | 根据 `/api/auth/menus` 注册 |
| 权限 Store | 保存权限标识集合 |
| 退出登录 | 清理 Token 和用户状态 |
| 403/404 | 中文提示页面 |

前端必须遵守 `docs/frontend-admin-spec.md`，不引入第二套 UI 或图标体系。

### 5.5 用户与角色管理

| 功能 | P0 要求 |
| --- | --- |
| 用户列表 | 分页、用户名/昵称/状态查询 |
| 新增用户 | 用户名、昵称、部门、角色、初始密码 |
| 编辑用户 | 修改基础信息、部门、状态 |
| 禁用用户 | 禁用后不可登录 |
| 重置密码 | 操作日志记录，不显示明文日志 |
| 分配角色 | 用户与角色关系可维护 |
| 角色列表 | 分页、角色编码/名称/状态查询 |
| 新增角色 | 角色编码唯一、数据范围、排序 |
| 分配菜单 | 树形勾选菜单和按钮权限 |

用户、角色、菜单、部门、字典和参数的编辑 DTO/VO 必须携带 version。两个会话并发编辑同一记录时，后提交者返回 `DATA_0409` 并提示刷新；分配角色、分配菜单在单个事务内保存目标集合，关系表组合唯一索引防止重复数据。普通创建不引入通用幂等键，前端提交期间禁用按钮，数据库唯一约束负责最终防重。

### 5.6 菜单与部门管理

| 功能 | P0 要求 |
| --- | --- |
| 菜单树 | 支持目录、菜单、按钮 |
| 菜单字段 | 名称、类型、路由、组件、权限标识、图标、排序、可见、状态 |
| 删除限制 | 有子菜单时禁止删除 |
| 部门树 | 支持父子结构、排序、状态 |
| 部门删除 | 有子部门或用户时禁止删除 |

菜单权限是后续代码生成的基础，权限标识必须遵守 `{domain}:{resource}:{action}`。

### 5.7 字典与参数配置

| 功能 | P0 要求 |
| --- | --- |
| 字典类型 | 编码唯一、名称、状态 |
| 字典项 | 标签、值、排序、状态 |
| 字典使用 | 前端可按 dictCode 查询并渲染 |
| 参数配置 | 普通系统参数管理 |
| 敏感限制 | `sys_config` 不保存 API Key、数据库密码、Redis 密码、TLS 私钥密码；P0 不使用 JWT Token Secret |

### 5.8 登录日志与操作日志

| 日志 | 要求 |
| --- | --- |
| 登录日志 | 成功/失败、用户名、用户 ID、IP、原因、时间 |
| 操作日志 | 模块、操作、路径、方法、状态、错误、操作人、时间 |
| 脱敏 | 不记录密码、Token、API Key、数据库密码 |
| 查询 | 支持分页和常用条件 |
| 删除 | P0 不提供日志删除 |

### 5.9 数据权限基础

| 能力 | P0 要求 |
| --- | --- |
| 数据范围枚举 | 全部、本部门、本部门及下级、仅本人 |
| 角色字段 | `sys_role.data_scope` |
| 用户上下文 | 当前用户携带 deptId 和角色数据范围 |
| 部门树 | 可计算下级部门 |
| 查询扩展点 | 为后续业务查询提供统一接入方式 |
| 生效状态说明 | API、页面或生成摘要能说明某模块数据权限是已接入还是仅定义 |

P0 不要求所有系统管理查询都实现复杂数据权限，但必须把表达模型和扩展点建立好。

### 5.10 本地文件基础服务

| 能力 | P0 要求 |
| --- | --- |
| 存储 | 只使用 `data/files` 本地目录，不引入 MinIO/OSS |
| 上传 | 单文件、20 MB、25 MB 请求、扩展名/MIME/内容三重校验 |
| 配额 | 默认 10 GB，保留至少 2 GB 磁盘空间，超限返回 `FILE_0507` |
| 白名单 | jpg/jpeg/png/webp/pdf/txt/md/csv/json；未列出的类型拒绝 |
| 下载 | 只按文件 ID 经权限接口下载，内部路径不进入响应 |
| 预览 | 仅白名单图片，其他类型返回 `FILE_0409` |
| 删除 | active/failed 经 deleting 到 deleted，失败进入 delete_failed 并可重试 |
| 中断恢复 | 启动时把超过 1 小时的 uploading 标记 failed，尽力清理同 storageKey 临时文件，不自动发布未知文件 |
| 审计 | 上传、下载、预览、删除和重试删除记录 traceId 与结果，日志脱敏 |
| 限制声明 | P0 无杀毒、无业务附件绑定、无分片/秒传、无公开链接、无 Office 在线预览 |

上传和删除的文件系统 I/O 不得包在长数据库事务中。必须覆盖路径穿越、伪造 MIME、超限流式上传、磁盘不足、原子移动失败、元数据写入失败和物理删除失败测试。

## 6. 后端交付物

| 模块 | 交付 |
| --- | --- |
| `vibe-security` | 登录态、当前用户、权限封装、数据范围扩展点 |
| `vibe-system` | 用户、角色、菜单、部门、字典、参数、日志 |
| `vibe-common` | 统一异常、分页、基础实体、脱敏工具 |

后端代码必须遵守 `docs/backend-implementation-spec.md`：Controller 薄、Service 控制事务、Mapper 简洁、DTO/VO 分离。

S2 在登录、角色和权限能力完成后实现 `/api/system/health`：要求 `system:health:info` 权限，并通过 Flyway 初始化对应菜单/权限；返回 `HEALTHY`、`DEGRADED`、`UNAVAILABLE` 三态及脱敏检查项；不得开放匿名详细诊断，不得返回主机名、连接串、数据库名、用户名、服务器绝对路径、配置值或异常堆栈。生产依赖完整性和脚本退出码仍由 S6 验收。

## 9. S2 禁止越界项

| 禁止 | 原因 |
| --- | --- |
| 做模型网关 | S3 范围 |
| 做 AI 工作台 | S4 范围；S3 只做模型网关 |
| 做代码生成器 | S4 范围 |
| 做客户拜访记录业务模块 | S4/S7 范围 |
| 做生产安装包 | S6 范围 |
| 做多租户 | P2 范围 |
| 做 OAuth/LDAP/企业微信登录 | P1/P2 范围 |
| 做工作流/报表 | P2 范围 |
| 引入新中间件 | 违反技术栈最小化 |
| 做业务附件、Office 在线预览、分片上传或对象存储 | 超出 P0 文件基础范围 |

## 10. 验证门禁

S2 完成后必须满足：

| 门禁 | 验收方式 |
| --- | --- |
| 后端完整验证 | `scripts/mvn.ps1 -pl vibe-starter -am test` 全部通过；关键登录、用户、角色、菜单、部门、字典、日志、文件接口有 MockMvc/API 正反用例，快速构建不得关闭 S2 |
| 前端构建 | `npm run build` |
| 登录成功 | 管理员能登录首页 |
| 登录失败 | 错误密码返回中文错误并记录日志 |
| 登录防爆破 | 账号/IP 双维度限流、通用失败提示、未知账号虚拟密码校验可验证 |
| 密码契约 | 15-128 code point、NFC、离线弱密码表和 PBKDF2 参数可验证，不强制字符组合或定期改密 |
| 会话 Cookie | Token 不进入响应体/Web Storage；HttpOnly、SameSite=Strict、Host-only 和 HTTPS Secure 属性可验证 |
| CSRF/Origin | 登录校验同源 Origin；已登录写请求缺少或伪造 CSRF Token 时被拒绝 |
| 会话撤销 | 当前登出、全部会话失效、最多 3 个会话和最早会话淘汰可验证 |
| 菜单权限 | 不同角色看到不同菜单 |
| 接口权限 | 无权限访问返回 `AUTH_0403` |
| 用户角色 | 用户可分配角色，角色可分配菜单 |
| 字典配置 | 字典类型和字典项可维护 |
| 操作日志 | 新增、修改、删除、重置密码有日志 |
| 数据权限基础 | 数据范围枚举、当前用户上下文、部门树、查询扩展点和未接入说明可验证 |
| 审计详情 | 操作日志包含 traceId、操作人、路径、目标对象、状态和脱敏错误摘要 |
| traceId 链路 | 统一响应体和 `X-Trace-Id` 一致，日志可按同一 traceId 定位，客户端不能覆盖服务端 traceId |
| 并发更新 | 两个会话编辑同一主记录，后提交者收到 `DATA_0409`，先提交内容不被覆盖 |
| 重复提交 | 唯一键冲突稳定返回 `DATA_0409`；关系保存和重复删除不产生重复记录或副作用 |
| 文件上传安全 | 大小、请求、配额、白名单、MIME/签名、路径穿越和内部路径隐藏可验证 |
| 文件访问控制 | 未登录或无权限不能列表、下载、预览或删除；不存在与无权限响应不泄漏内部路径 |
| 文件数据范围声明 | P0 是 RBAC 文件管理权限，不宣称已有业务附件所有权或部门级隔离 |
| 文件删除状态 | 删除成功物理文件不可访问；删除失败进入 delete_failed 且可受控重试 |

## 11. AI 实现提示

外部 AI Coding 工具执行 S2 时必须遵守：

| 规则 | 说明 |
| --- | --- |
| 先读文档 | `docs/README.md`、本文、`basic-admin-spec.md`、后端/前端/数据库规范 |
| 小步提交 | 一次只做一个任务组，避免系统管理和前端布局混改过大 |
| 保持权限同源 | 菜单、按钮、接口权限标识一致 |
| 不写死生产密码 | 开发默认账号不能进入生产安装策略 |
| 不绕过首次改密 | 初始化管理员未改密前不能访问菜单、业务页面或 AI 工作台 |
| 生成后验证 | 按 `quality-gates.md` 输出验证摘要 |
| 发现冲突先改文档 | 不用代码绕过未决策问题 |

## 12. 完成定义

S2 只有在以下条件同时满足时才算完成：

| 条件 | 说明 |
| --- | --- |
| 功能完成 | 登录、用户、角色、菜单、部门、字典、参数、日志和本地文件基础服务可用 |
| 权限闭环 | 前端按钮、菜单和后端接口权限一致 |
| 数据基础可用 | 初始数据、字典、部门、数据范围基础可用 |
| 审计可查 | 登录和关键操作有日志 |
| 权限说明不误导 | 未完全接入数据权限的模块必须在摘要或页面说明限制 |
| 验证通过 | 后端和前端构建/测试达到 S2 门禁 |
| 文档同步 | 若实现改变字段、API、权限或范围，相关文档已更新 |

## 13. 一句话总结

S2 基础后台要把企业系统的“门、钥匙、人员、组织、字典和审计账本”先建稳。只有这层稳定，后面的 AI 工作台和代码生成才不会在权限、菜单和日志上反复补洞。

# Vibe Boot AI 工具使用指南

## 1. 文档目的

本文定义 Vibe Boot 用户在开发模式、平台内 AI 工作台和生产模式下如何使用 AI 工具。

`docs/ai-tooling-strategy.md` 解决“系统分工是什么”，本文解决“用户拿到项目后具体怎么用 AI”。两者共同约束 MVP：Vibe Boot 首版不自研完整 AI IDE，而是把外部 AI Coding 工具、平台内 AI 工作台、模型网关、skills 和质量门禁组合成一个可落地的开发闭环。

## 2. 核心结论

| 结论 | 说明 |
| --- | --- |
| 开发模式优先使用外部 AI Coding 工具 | Codex、Cursor、Claude Code、通义灵码等工具负责真实源码修改 |
| 平台内 AI 工作台负责受控流程 | 面向企业用户做需求澄清、计划、风险、生成、验证摘要 |
| 生产模式禁止开发型 AI 能力 | 生产包不包含改源码、执行脚本或改数据库结构的页面、API、路由和处理器，不能由配置恢复 |
| 所有 AI 都必须读取项目上下文 | `docs/README.md`、ADR、产品约束、模块规范、质量门禁是默认入口 |
| 生成代码必须经过验证 | 后端编译、前端构建、必要测试和中文失败说明是最低要求 |

一句话：

> 用户不是从空白仓库开始 vibe coding，而是在 Vibe Boot 准备好的 Java 单体底座、文档、规则和验证链路上使用 AI 持续迭代。

## 2.1 待签收的使用模型口径

“如何使用 AI 工具”在 MVP 阶段已形成以下文档口径。它仍需通过 `docs/coding-start-signoff.md` 人工签收；签收后进入编码阶段时，不得重新解释成另一套产品形态。

| 问题 | 冻结答案 | 产品含义 |
| --- | --- | --- |
| Vibe Boot 是否替代 Codex、Cursor、Claude Code | 否 | 首版不自研完整 AI IDE，不和成熟 AI Coding 工具竞争 |
| 用户是否必须从零写代码 | 否 | 用户从 Vibe Boot 已准备好的 Java/Vue 单体底座开始迭代 |
| 企业用户是否必须理解源码 | 否 | 企业用户通过平台 AI 工作台表达需求、确认计划和查看结果 |
| 谁负责真实文件修改 | P0 通用源码修改由外部 AI Coding 工具负责；平台确定性生成器只写其声明拥有的生成路径 | P1 才可另行设计本地受控补丁执行器，生产始终禁止 |
| 平台自己的 AI 价值是什么 | 把需求、上下文、规则、风险、确认和验证流程产品化 | 平台不是 Chat 外壳，而是受控生成流程 |
| 生产环境是否还能 vibe coding | 否 | 生产只运行构建产物，只保留安全业务 AI |

首版使用模型：

| 层次 | 入口 | 用户感知 |
| --- | --- | --- |
| 工程执行层 | Codex、Cursor、Claude Code、通义灵码等 | 修改代码、运行命令、修复错误 |
| 平台产品层 | Vibe Boot AI 工作台 | 填需求、答澄清、看计划、确认风险、看验证摘要 |
| 模型调用层 | Vibe Boot 模型网关 | 配模型、测连接、看用量和错误 |
| 生产业务层 | 已部署业务 AI 功能 | 摘要、问答、文案、分析，不涉及代码 |

最短使用路径固定为：

| 用户身份 | 第一步 | 第二步 | 第三步 |
| --- | --- | --- | --- |
| 企业管理员 | 打开平台 AI 工作台 | 描述业务需求并确认澄清问题 | 查看计划、风险、预览和验收结果 |
| 实施人员 | 从工作台导出外部 AI 交接包 | 交给 Codex/Cursor/Claude Code 等执行 | 复核文件清单、验证结果和风险摘要 |
| Java 开发者 | 直接用外部 AI Coding 工具打开仓库 | 读取 `docs/README.md` 和阶段文档 | 小步修改、验证并回填摘要 |
| 生产用户 | 使用生产业务 AI 页面 | 执行业务问答、摘要、分类、文案或分析 | 不接触源码、脚本和安装目录 |

## 2.2 签收前必须确认的 AI 使用口径

本文给出的是编码前的产品约束口径。它可以作为签收依据，但在 `docs/coding-start-signoff.md` 未签收前，不能被解释为已经允许实现 AI 工作台、代码生成器或源码目录。

| 确认项 | 建议口径 | 不接受时的处理 |
| --- | --- | --- |
| 是否首版自研完整 AI IDE | 不做 | 回到 `docs/adr/0003-ai-tool-usage-boundary.md` 重开决策 |
| 企业用户是否必须直接使用 Codex/Cursor/Claude Code | 不必须 | 强化平台 AI 工作台和实施人员交接路径 |
| 外部 AI Coding 工具是否是开发模式真实源码修改主路径 | 是 | 需要重新设计 P0 Patch Manager 和安全边界 |
| 平台 AI 工作台是否只是普通 Chat | 否 | 必须围绕需求、上下文、计划、风险、交接包和验证摘要工作 |
| 平台是否替用户隐藏全部工程复杂度 | 否 | 企业用户不必懂源码，但实施人员或开发者仍需能接管真实代码 |
| 交接包是否可以在生产执行 | 否 | 生产只能接收受控安装包、迁移和回滚流程 |
| 生产模式是否允许开发型 AI | 否 | 如需生产 AI 运维，必须新增安全设计和签收流程 |

签收时要确认的不是“以后永远不做更强的 AI”，而是确认 MVP 阶段先采用外部 AI Coding 工具 + 平台 AI 工作台 + 模型网关 + 生产业务 AI 这条低复杂度路径。

## 2.3 AI 工具使用定稿

“AI 工具怎么用”在 MVP 阶段按下表定稿。后续如果要改变其中任一项，必须先更新 ADR、产品约束、任务分解和签收材料。

| 决策点 | MVP 定稿 | 直接影响 |
| --- | --- | --- |
| 是否内置完整 AI Coding Agent | 不内置 | 不实现 IDE、完整终端、Git Agent、长期后台 Agent |
| 是否替代 Codex/Cursor/Claude Code | 不替代 | 平台适配和约束外部工具，而不是重做它们 |
| 企业用户从哪里开始 | 平台 AI 工作台 | 企业用户描述业务、回答澄清、确认计划和验收结果 |
| 实施人员或开发者从哪里开始 | 外部 AI Coding 工具 | 使用交接包在开发工作区修改源码、运行验证、修复问题 |
| 外部 AI 工具是否是黑盒 | 不是 | 必须读取文档、遵守范围、输出文件清单、验证结果和风险摘要 |
| 平台工作台是否直接写生产 | 不写 | 工作台产物进入生产只能通过开发验证、生产包和升级流程 |
| 生产模型配置是否恢复开发能力 | 不恢复 | 生产模型只允许业务问答、摘要、分类、文案、分析和连接测试 |

首版 AI 工具使用链路固定为：

| 顺序 | 责任方 | 产物 | 不允许 |
| --- | --- | --- | --- |
| 1 | 企业用户 | 业务需求、字段、菜单、权限、验收意见 | 被要求直接理解源码目录和构建命令 |
| 2 | 平台 AI 工作台 | 澄清记录、变更计划、风险、元模型、外部 AI 交接包 | 变成普通聊天框或万能执行器 |
| 3 | 实施人员/开发者 | 将交接包交给外部 AI Coding 工具并复核结果 | 不看验证结果就交付 |
| 4 | 外部 AI Coding 工具 | 源码变更、验证结果、风险摘要 | 越过文档和阶段边界自由发挥 |
| 5 | 开发模式脚本 | 构建、测试、预览、生产包生成 | 把补丁或临时 SQL 直接拿到生产运行 |
| 6 | 生产安装包 | 安装、升级、备份、恢复、业务 AI | 在线改源码、执行 shell、执行交接包 |

这个定稿解决的不是“选哪一家 AI 工具”，而是让任何 AI 工具都必须在 Vibe Boot 的文档、任务、风险和验证轨道内工作。

## 3. AI 工具类型

| 工具类型 | 示例 | 首版角色 | 是否内置 |
| --- | --- | --- | --- |
| 外部 AI Coding 工具 | Codex、Cursor、Claude Code、GitHub Copilot、通义灵码 | 修改源码、运行命令、补测试、解释工程 | 不内置，文档适配 |
| 平台内 AI 工作台 | Vibe Boot 管理端 AI 页面 | 需求澄清、计划、风险、受控生成、验证摘要 | 内置 |
| 模型网关 | OpenAI Compatible、国内大模型 API | 模型配置、调用、用量、脱敏、审计 | 内置 |
| Skills/规则 | Markdown skills、项目约束、生成规则 | 约束 AI 行为和代码风格 | 内置项目上下文 |
| 生产业务 AI | 摘要、分类、问答、文案、数据分析 | 辅助业务使用，不参与代码修改 | 可选开启 |

## 3.1 首版推荐入口

用户不需要先决定“到底用哪个 AI 产品才能开发”。Vibe Boot 的口径是：不同用户角色走不同入口，但共享同一套文档和质量门禁。

| 用户角色 | 推荐入口 | 典型任务 | 需要理解源码吗 | 约束 |
| --- | --- | --- | --- | --- |
| 项目维护者 | 外部 AI Coding 工具 | 搭工程骨架、修 Bug、补测试、维护文档 | 需要 | 必须读 `docs/README.md` 并按阶段任务施工 |
| Java 全栈开发者 | 外部 AI Coding 工具 + IDE | 接管 AI 生成代码、扩展复杂业务 | 需要 | 不绕过 ADR、模块边界和质量门禁 |
| 实施人员 | 平台 AI 工作台 + 外部 AI Coding 工具 | 把客户需求落成业务模块 | 部分需要 | 工作台沉淀需求和计划，外部工具处理源码 |
| 企业管理员 | 平台 AI 工作台 | 配置模型、确认需求、预览生成结果 | 不强制 | 高风险操作必须确认 |
| 业务用户 | 生产业务 AI | 摘要、问答、文案、分析 | 不需要 | 不能触发代码修改、脚本执行和结构变更 |

首版默认路径：

| 阶段 | 主入口 | 原因 |
| --- | --- | --- |
| S1-S2 工程与基础后台 | 外部 AI Coding 工具 | 主要是源码工程建设，成熟工具效率最高 |
| S3 模型网关 | 外部 AI Coding 工具实现，管理端提供配置入口 | 先把模型调用治理住 |
| S4 AI 工作台和代码生成 | 平台 AI 工作台成为主用户入口 | 企业用户开始通过工作台描述业务 |
| S5-S6 开发包和生产包 | 脚本 + 外部 AI Coding 工具维护 | 构建、安装、验证需要真实命令闭环 |
| S7 演示验收 | 平台 AI 工作台 + 生产安装包 | 证明用户能从需求走到可运行系统 |

## 3.2 AI 工具选择矩阵

用户不需要在每次任务前重新判断工具边界。按任务类型选择默认入口：

| 任务类型 | 默认入口 | 备用入口 | 禁止入口 |
| --- | --- | --- | --- |
| 修改项目源码 | 外部 AI Coding 工具 | P1 本地受控补丁执行器 | 生产服务器 |
| 生成业务模块需求计划 | 平台 AI 工作台 | 外部 AI Coding 工具辅助整理 | 普通 Chat 无项目上下文 |
| 应用代码补丁 | 外部 AI Coding 工具 | P1 Patch Manager | P0 服务端任意 shell |
| 运行 Maven/npm/PowerShell 验证 | 外部 AI Coding 工具或本地脚本 | P1 Verification Runner | 生产业务页面 |
| 配置模型供应商 | 平台模型配置页面只写录入 API Key并加密入库 | `setup-model.ps1`/安装脚本只初始化外置主密钥 | 业务模块写死 API Key、配置文件保存供应商 API Key |
| 生产业务问答或摘要 | 生产业务 AI 页面 | 后台管理端受控入口 | 开发型 AI 工具 |
| 生产升级和回滚 | 生产安装脚本 | 人工运维 | AI 在线修改源码 |

选择原则：

| 原则 | 说明 |
| --- | --- |
| 改代码回开发模式 | 任何源码、迁移脚本、前端资源修改都必须回开发模式完成 |
| 企业用户走工作台 | 不要求企业用户直接组织 Prompt、文件列表和验证命令 |
| 实施人员做桥接 | 实施人员可以把工作台输出的需求计划交给外部 AI Coding 工具执行 |
| 生产只做业务 AI | 生产模型能力只能服务业务，不承担开发工具职责 |

## 3.3 AI 工具使用责任边界

Vibe Boot 不把“会不会用 AI 工具”留给用户自行摸索。首版必须把 AI 工具的责任边界写进产品、文档和交接包。

| 问题 | 标准答案 | 产品约束 |
| --- | --- | --- |
| 企业用户是否直接使用 Codex/Cursor | 不要求 | 企业用户主要使用平台 AI 工作台描述需求、确认计划和验收结果 |
| 实施人员是否需要接触外部 AI Coding 工具 | 需要 | 实施人员负责把平台交接包交给外部 AI Coding 工具，并复核结果 |
| Java 开发者是否仍然有价值 | 有 | 复杂问题、框架升级、安全变更和疑难错误需要开发者接管 |
| 平台是否承诺一键自动完成所有编码 | 不承诺 | P0 承诺需求到交接包和可审查生成闭环，不承诺完整无人值守开发 |
| AI 工具是否可以自行决定技术方案 | 不可以 | 技术栈、目录、依赖、阶段边界和验证命令以文档为准 |
| 工作台是否可以替代外部 AI Coding 工具 | P0 不替代 | 工作台是产品化入口，外部 AI Coding 工具是源码执行入口 |

角色之间的交接必须可追溯：

| 交接方向 | 交接物 | 验收口径 |
| --- | --- | --- |
| 企业用户 -> AI 工作台 | 自然语言需求、澄清回答、风险确认 | 工作台形成结构化需求和变更计划 |
| AI 工作台 -> 实施人员 | 外部 AI 交接包、准入卡、验证要求 | 实施人员能直接复制给外部 AI Coding 工具 |
| 实施人员 -> 外部 AI Coding 工具 | 受控任务提示词和允许范围 | 外部 AI 只在开发工作区内修改源码 |
| 外部 AI Coding 工具 -> 实施人员 | 文件清单、验证结果、风险摘要 | 实施人员可判断是否交给企业用户预览 |
| 实施人员 -> 企业用户 | 可预览页面、变更说明、剩余限制 | 企业用户按业务目标验收，不需要阅读源码 |

## 4. 开发者使用路径

开发者或实施人员可以直接使用外部 AI Coding 工具打开 Vibe Boot 仓库。

当前未签收时，本节只能用于指导文档修订、上下文读取和任务判断；不得据此创建源码目录或修改非文档文件。

| 步骤 | 用户动作 | AI 应做 | Vibe Boot 提供 |
| --- | --- | --- | --- |
| 1 | 打开仓库 | 读取 `docs/README.md` | 文档索引和编码闸门 |
| 2 | 选择任务 | 判断任务属于 S1-S7 哪一阶段 | `mvp-roadmap.md` 和任务分解 |
| 3 | 读取约束 | 读取相关 ADR、规格和规范 | 产品约束、模块设计、质量门禁 |
| 4 | 形成计划 | 输出文件、接口、数据库、前端影响 | 统一计划口径 |
| 5 | 修改代码 | 使用 patch 或受控编辑 | 真实 Java/Vue/SQL 工程 |
| 6 | 执行验证 | 运行 Maven、npm、脚本检查 | `quality-gates.md` |
| 7 | 输出摘要 | 说明变更、验证、风险、下一步 | 中文交付格式 |

开发者路径的默认规则：

| 规则 | 说明 |
| --- | --- |
| 先文档后代码 | 需求边界不清时先改文档 |
| 先小任务后大任务 | 不跨阶段混写 S1、S2、S3 |
| 先验证后交付 | 不能只说“已修改”而不说明验证结果 |
| 不自动提交推送 | Git commit 和 push 必须由用户确认 |
| 不覆盖用户改动 | 修改前必须检查当前工作区状态 |

## 4.1 外部 AI Coding 最小闭环

外部 AI Coding 工具每次处理真实源码任务时，最低必须完成以下闭环。

| 阶段 | 必须产出 | 不合格表现 |
| --- | --- | --- |
| 读取上下文 | 说明已读取的文档、任务阶段和限制 | 只根据用户一句话直接改代码 |
| 计划 | 文件清单、接口影响、数据库影响、前端影响、验证计划 | 只说“我会实现这个功能” |
| 修改 | 小步 patch，保持阶段边界 | 跨阶段顺手实现未授权功能 |
| 验证 | Maven、npm、脚本或明确的未验证原因 | 没有验证就宣称完成 |
| 摘要 | 变更内容、风险、验证结果、下一步 | 只输出“已完成” |

推荐一轮任务的最短提示词：

```text
请先读取 docs/README.md，并判断当前任务属于哪个阶段。
只处理本次指定任务；如果需求、技术栈或阶段边界不清，先修订 docs/，不要直接编码。
修改后请输出文件清单、验证命令、验证结果和剩余风险。
```

## 4.2 开发模式首次使用 AI 工具

开发模式不能假设用户已经会组织 AI coding 上下文。用户拿到项目后的首次 AI 使用路径必须固定为下面的产品动作。

| 顺序 | 用户动作 | 平台或文档必须提供 | AI 工具职责 |
| --- | --- | --- | --- |
| 1 | 解压 Windows 开发包 | 中文入口说明、目录结构说明 | 不参与 |
| 2 | 执行环境检查 | `scripts/doctor.ps1`、JDK/Maven/Node/MySQL/Redis 检查结果 | 解释失败原因和修复建议 |
| 3 | 配置国内镜像 | Maven `settings.xml`、npm registry 配置说明 | 不自行更换技术栈或包管理器 |
| 4 | 启动开发模式 | `scripts/dev-start.ps1`、访问地址、日志路径 | 根据日志定位启动失败 |
| 5 | 配置模型 | 模型配置页面、连接测试、错误中文化 | 不把 API Key 写入源码或提交 Git |
| 6 | 读取项目上下文 | `docs/README.md`、ADR、任务分解、质量门禁 | 先说明读取了哪些文档 |
| 7 | 选择任务来源 | 工作台任务、阶段任务单或维护者自然语言指令 | 判断是否允许编码，不清楚时先改文档 |
| 8 | 开始一轮修改 | 计划、风险、文件清单、验证命令 | 小步修改、验证、摘要 |

首次使用时，平台必须让用户明白一件事：AI 不是脱离项目规则的聊天助手，而是在 Vibe Boot 已准备好的工程、文档、skills 和脚本约束下工作的开发协作者。

首次把任务交给外部 AI Coding 工具时，推荐使用下面的最小开场话术。该话术不是新的授权来源；如果签收状态不允许编码，AI 工具仍只能修订文档。

```text
你正在处理 Vibe Boot 项目。
请先读取 docs/README.md、docs/product-constraints.md、docs/ai-tool-usage-guide.md 和当前任务相关阶段文档。

当前目标：
当前阶段：
允许修改范围：
禁止事项：
预期验证命令：

请先输出 AI 使用准入卡结论、读取到的关键约束和执行计划。
如果当前签收状态或启动口令不允许编码，只能修订 docs/，不得创建源码目录。
```

平台 AI 工作台在 P0 阶段必须能帮助用户生成这段内容，而不是只给出一句“请使用 AI 工具继续开发”的空泛提示。

## 4.3 外部 AI 工具交接包

平台 AI 工作台生成的计划如果需要交给 Codex、Cursor、Claude Code、通义灵码等外部 AI Coding 工具执行，必须形成一个可复制的交接包。

| 内容 | 必填 | 说明 |
| --- | --- | --- |
| 任务标题 | 是 | 一句话说明要做什么 |
| 当前阶段 | 是 | S1-S7 或后续阶段，禁止跨阶段偷跑 |
| 业务目标 | 是 | 用户真正想解决的业务问题 |
| 需求澄清结论 | 是 | 字段、权限、菜单、数据范围、边界 |
| 允许修改范围 | 是 | 目录、模块、页面、SQL、测试范围 |
| 禁止事项 | 是 | 不新增依赖、不改安全核心、不碰生产配置等 |
| 风险等级 | 是 | L0-L3，并说明是否已确认 |
| 预期验证命令 | 是 | Maven、npm、PowerShell 或明确的免测原因 |
| 相关文档 | 是 | 至少包含 `docs/README.md` 和阶段任务文档 |
| 输出格式 | 是 | 文件清单、验证结果、风险、后续建议 |

推荐交接包模板：

```text
你正在处理 Vibe Boot 的一个受控开发任务。

请先阅读 docs/README.md、docs/product-constraints.md、docs/ai-tool-usage-guide.md 和本任务关联的阶段文档。

任务标题：
当前阶段：
业务目标：
需求澄清结论：
允许修改范围：
禁止事项：
风险等级：
预期验证命令：

要求：
1. 如果发现签收状态不允许编码，只能修订 docs/。
2. 如果任务边界不清，先补文档，不要直接写代码。
3. 修改后输出文件清单、验证命令、验证结果和剩余风险。
```

交接包的目标是降低“会不会用 AI 工具”的门槛：企业用户负责确认业务，平台负责组织上下文，实施人员或开发者负责把交接包交给外部 AI Coding 工具执行。

## 4.3.1 交接包不是授权书

外部 AI 交接包只是执行上下文，不是新的编码授权来源。它必须同时满足签收状态、阶段启动口令、任务范围和质量门禁，才能进入源码修改。

| 判断 | 允许继续 | 必须停止 |
| --- | --- | --- |
| 签收状态 | `coding-start-signoff.md` 已签收且允许对应阶段编码 | 未签收、只给了交接包、只说“继续” |
| 启动口令 | 维护者给出对应阶段的精确启动口令 | 口令缺失、带额外条件、阶段不匹配 |
| 执行位置 | 开发工作区 | 生产服务器、生产安装目录、线上数据库 |
| 修改范围 | 交接包列明的目录、模块和文件类型 | 顺手修改依赖、权限核心、发布脚本或跨阶段模块 |
| 验证输出 | 有命令、结果、失败说明和风险摘要 | 只输出“已完成” |

任何 AI 工具如果无法证明上述判断通过，只能要求补充文档或澄清任务，不能自行开始编码。

## 4.4 AI 使用准入卡

每次把任务交给 AI 前，必须先完成以下准入判断。准入卡的目标不是增加流程，而是防止把未签收、跨阶段、无验证或生产高风险请求伪装成普通 AI 对话。

| 判断项 | 通过标准 | 不通过时动作 |
| --- | --- | --- |
| 当前是否允许编码 | `docs/coding-start-signoff.md` 已签收，且维护者已发出对应阶段启动口令 | 只允许修订 `docs/`，不得创建或修改源码目录 |
| 任务阶段是否明确 | 能归入 S1-S7 或后续已签收阶段，且不跨阶段偷跑 | 回到路线图、任务分解或 ADR 补充约束 |
| 执行入口是否正确 | P0 通用源码修改走外部 AI Coding 工具，确定性生成只写 owned 路径，企业需求走 AI 工作台，生产只走业务 AI | 重新选择入口，禁止在生产服务器承接开发动作 |
| 上下文是否完整 | 至少包含 `docs/README.md`、产品约束、相关 ADR、阶段任务和质量门禁 | 补齐交接包或阅读顺序后再继续 |
| 风险是否已标注 | 已说明权限、数据、依赖、部署、迁移、回滚和验证影响 | 标记风险并请求人工确认 |
| 验证是否可执行 | 有 Maven、npm、PowerShell、测试或明确免测原因 | 先补验证计划，不能只输出“已完成” |
| 生产边界是否安全 | 不包含在线改源码、执行 shell、直接改表或执行外部 AI 交接包 | 拒绝执行，回到安全治理和生产包设计 |

准入卡结论必须出现在外部 AI 交接包、平台 AI 工作台任务详情或 AI 工具输出摘要中。没有准入卡结论的任务，不能被视为已经满足 Vibe Boot 的 AI 使用约束。

## 5. 企业用户使用路径

企业用户不一定直接操作 IDE。MVP 应让用户主要通过 Vibe Boot 管理端 AI 工作台完成业务模块生成。

| 步骤 | 用户动作 | 平台反馈 |
| --- | --- | --- |
| 1 | 解压 Windows 开发包 | 提供中文 README 和脚本入口 |
| 2 | 执行 `scripts/doctor.ps1` | 检查 JDK、Maven、Node、MySQL、Redis、端口、镜像 |
| 3 | 执行 `scripts/dev-start.ps1` | 启动后端、前端和必要开发服务 |
| 4 | 登录管理端 | 进入基础后台 |
| 5 | 配置大模型 | 测试 API Base、API Key、模型名 |
| 6 | 打开 AI 工作台 | 展示任务输入和上下文说明 |
| 7 | 描述需求 | 例如“帮我做客户拜访记录模块” |
| 8 | 回答澄清问题 | 确认字段、权限、菜单、数据范围 |
| 9 | 确认变更计划 | 查看风险、文件、表、接口、页面 |
| 10 | 执行生成与验证 | 查看生成结果、diff 摘要和验证结果 |
| 11 | 本地预览 | 在管理端试用新增模块 |
| 12 | 生成生产包 | 执行 `scripts/build-prod.ps1` |

企业用户视角下，AI 使用不暴露成“打开 IDE 自己写代码”，而是下面的产品动作：

| 用户看到 | 平台内部对应 |
| --- | --- |
| 描述业务需求 | 生成结构化需求和澄清问题 |
| 回答字段和权限问题 | 形成元模型、权限、菜单和数据范围 |
| 确认生成计划 | 记录风险、文件、SQL、接口和页面影响 |
| 查看生成结果 | 展示 diff 摘要、验证摘要和可预览页面 |
| 决定是否打包生产 | 进入 S6 构建、安装、备份和恢复流程 |

如果企业用户没有开发能力，首版默认由实施人员或项目维护者使用外部 AI Coding 工具执行源码修改；平台负责把需求和计划沉淀清楚，避免实施人员从空白口述重新理解业务。

## 5.1 企业用户不直接使用外部 AI 工具时的路径

首版必须承认一个现实：很多中小企业用户不会直接打开 IDE 或 Agent 终端。Vibe Boot 不能把“用户会自己 vibe coding”当作前提。

| 阶段 | 企业用户看到 | 实施人员或维护者看到 | 平台必须沉淀 |
| --- | --- | --- | --- |
| 需求输入 | 自然语言业务描述 | 结构化需求草稿 | 原始需求和澄清记录 |
| 计划确认 | 字段、菜单、权限、风险 | 可执行任务清单 | 交接包、风险等级、确认记录 |
| 代码执行 | 等待生成和验证结果 | 外部 AI Coding 工具修改源码 | 文件清单、diff 摘要、验证摘要 |
| 预览验收 | 在管理端试用页面 | 修复缺陷或补测试 | 验收意见和问题记录 |
| 打包发布 | 生成生产包 | 执行构建、安装、备份检查 | 发布记录和回滚点 |

因此，平台 AI 工作台的 P0 价值不是“在线帮用户直接写完整系统”，而是把业务语言转换为外部 AI Coding 工具可以执行的工程语言。

## 5.2 企业用户不会 AI 工具时的默认方案

当企业用户不知道 Codex、Cursor、Claude Code 或通义灵码是什么时，平台不能把任务退回给用户。默认方案是“工作台托底、实施人员接力、外部工具执行”。

| 用户问题 | 平台默认回应 | 后台形成的工程材料 |
| --- | --- | --- |
| 我不会用 AI Coding 工具 | 先在工作台描述需求，平台引导澄清和确认 | 结构化需求、字段、权限、菜单、风险 |
| 我不知道要给 AI 什么上下文 | 工作台自动引用文档入口和阶段任务 | 相关文档列表、准入卡、允许范围 |
| 我不会判断 AI 改得对不对 | 平台展示预览、验证摘要和风险说明 | 文件清单、验证命令、失败原因、剩余风险 |
| 我没有开发人员 | 先生成交接包和演示预览；复杂问题提示需要实施人员 | 可复制交接包、实施任务单、验收清单 |
| 我想直接上线 | 引导先生成生产包并走安装/升级流程 | 构建记录、版本、迁移、备份和回滚说明 |

该路径的产品承诺是降低 AI 使用门槛，不是消灭所有专业判断。涉及安全、数据、权限、复杂业务规则或生产发布时，仍必须有实施人员或开发者复核。

## 5.3 AI 工具能力成熟度分层

为了避免首版过度承诺，AI 工具能力按成熟度分层交付。

| 层级 | 名称 | 首版状态 | 用户承诺 |
| --- | --- | --- | --- |
| A0 | 文档化 AI 使用 | 必做 | 提供阅读顺序、提示词、任务单和验证要求 |
| A1 | 工作台需求与计划 | 必做 | 用户能在平台内沉淀需求、澄清和变更计划 |
| A2 | 外部 AI 执行交接 | 必做 | 平台能输出给外部 AI Coding 工具的交接包 |
| A3 | 本地受控补丁应用 | P1 | 平台可受控应用补丁，但仍在开发模式 |
| A4 | 后台验证 Runner | P1 | 平台可调度 Maven/npm/脚本验证并记录结果 |
| A5 | 完整内置 Agent IDE | 不做 | 首版不承诺 |
| A6 | 生产环境开发 Agent | 禁止 | 生产永不作为默认开发入口 |

A0-A2 是 MVP 必须满足的 AI 使用闭环；A3-A4 是增强体验；A5-A6 不是首版目标。

## 6. 平台内 AI 工作台 P0 边界

P0 的 AI 工作台必须轻，不做完整 IDE。

| 能力 | P0 状态 | 说明 |
| --- | --- | --- |
| 模型配置 | 必做 | 供应商、API Base、API Key、模型名 |
| 模型连接测试 | 必做 | 中文展示失败原因 |
| 需求澄清 | 必做 | 字段、权限、数据影响必须问清楚 |
| 变更计划 | 必做 | 文件、接口、SQL、菜单、页面、测试计划 |
| 风险分级 | 必做 | L0-L3 风险 |
| 元模型生成 | 必做 | 单表 CRUD 所需结构化模型 |
| 代码补丁生成 | 必做 | P0 由外部 AI Coding 工具应用；P1 本地受控执行器需另行签收 |
| diff 展示 | 基础支持 | P0 可纯文本，P1 再做专业 diff viewer |
| 后台执行终端 | 暂不做 | 避免服务端任意命令执行风险 |
| 完整 Agent Runtime | 暂不做 | P0 依赖外部 AI Coding 工具 |

## 7. 外部 AI Coding 工具适配要求

Vibe Boot 需要主动适配外部 AI Coding 工具，而不是假设用户会自己组织上下文。

| 适配项 | 要求 |
| --- | --- |
| 文档入口 | `docs/README.md` 必须列出阅读顺序和编码闸门 |
| 任务文档 | 每个阶段应有任务分解，便于 AI 小步施工 |
| 规则显式化 | 技术栈、禁止事项、质量门禁必须写成清单 |
| 命令标准化 | Maven、npm、PowerShell 验证命令必须固定 |
| 错误可解释 | 文档要求 AI 输出中文失败摘要 |
| 变更可审查 | AI 必须输出文件清单、风险和验证结果 |

推荐外部 AI 工具开场提示：

```text
请先阅读 docs/README.md，并按文档优先原则工作。
本项目首版是 Windows 优先的 Java/Vue 模块化单体。
不要扩大技术栈，不要跳过质量门禁，不要在需求不清时直接编码。
当前任务只处理指定阶段和指定文档/模块。
```

更完整的外部 AI Coding 工具提示词、S1 开工提示和输出摘要模板见 `docs/external-ai-coding-prompt.md`。

## 8. 开发模式与生产模式差异

| 能力 | 开发模式 | 生产模式 |
| --- | --- | --- |
| 外部 AI Coding 工具 | 可使用 | 不在生产服务器使用 |
| 平台内 AI 工作台 | 可做需求、计划、生成、验证 | 默认只做问答/解释类能力 |
| 代码修改 | 允许，但必须审查和验证 | 禁止 |
| 脚本执行 | 允许开发脚本 | 禁止开发型脚本 |
| 数据库结构变更 | 允许迁移脚本，需确认 | 禁止在线结构修改 |
| 模型配置 | 可配置开发模型 | 可配置业务模型 |
| 密钥保存 | 本地 ignored 配置或加密存储 | 生产配置独立保存 |

## 9. 中国用户体验约束

AI 工具使用体验必须考虑国内网络和 Windows 环境。

| 项目 | 约束 |
| --- | --- |
| Maven | 默认国内镜像，提供项目级 `settings.xml` |
| npm | 默认 `https://registry.npmmirror.com` |
| Node | 开发包预置 Node.js 24.x LTS（基线 24.18.0） |
| JDK | 开发包预置或自动定位 JDK 17 |
| 错误提示 | 依赖下载失败、模型连接失败必须中文提示 |
| 模型供应商 | OpenAI 兼容协议优先，支持国内常见模型 |
| 文档语言 | 首版文档和工作台提示中文优先 |

## 10. 风险确认规则

| 风险 | 示例 | 处理方式 |
| --- | --- | --- |
| L0 | 文档、注释、小样式 | 可直接修改并摘要 |
| L1 | 新增普通 CRUD | 计划后执行，必须验证 |
| L2 | 新增依赖、改表结构、权限菜单变更 | 必须用户确认 |
| L3 | 删除数据、修改登录鉴权、生产发布 | 必须二次确认，建议备份 |

AI 工具不得把“用户说继续”理解成允许执行所有高风险操作。风险确认必须针对具体动作。

## 11. 上下文读取顺序

无论是外部 AI Coding 工具，还是平台内 AI 工作台，都应按以下顺序读取上下文：

| 顺序 | 上下文 | 用途 |
| --- | --- | --- |
| 1 | `docs/README.md` | 找到阅读顺序和编码闸门 |
| 2 | ADR | 确认技术选型和实现契约 |
| 3 | `product-constraints.md` | 确认做什么、不做什么 |
| 4 | 阶段任务分解 | 确认当前任务边界 |
| 5 | 模块/后端/前端/API/数据库规范 | 确认实现方式 |
| 6 | `quality-gates.md` | 确认验证命令 |
| 7 | 相关源码 | 执行具体修改 |

## 12. 编码准入影响

在“AI 工具怎么用”没有确认前，不应进入 AI 工作台或代码生成相关编码。本文给出的准入结论是：

| 项目 | 结论 |
| --- | --- |
| 是否自研完整 AI IDE | P0 不做 |
| 是否强依赖外部 AI Coding 工具 | 开发实现阶段是主路径 |
| 是否内置 AI 工作台 | 是，但只做受控流程 |
| 是否允许生产改代码 | 否 |
| 是否要求 AI 读取文档 | 是，`docs/README.md` 是入口 |
| 是否要求验证 | 是，遵守 `docs/quality-gates.md` |

## 13. 一句话总结

Vibe Boot 的 AI 工具使用方式是：开发模式借助外部 AI Coding 工具完成真实源码演进，平台内 AI 工作台把高频业务生成流程产品化，生产模式只保留安全的业务 AI 能力，所有 AI 行为都受文档、skills、风险确认和质量门禁约束。

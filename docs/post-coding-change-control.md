# Vibe Boot 编码后变更控制

## 1. 文档目的

本文定义 Vibe Boot 在正式进入 S1 工程骨架编码之后，如何继续执行文档优先、范围控制、风险确认和质量门禁。

它不是新的需求来源，也不是放宽编码许可。它的作用是防止项目一旦开始编码，就把“先写文档”误解为已经结束。

## 2. 核心结论

| 结论 | 说明 |
| --- | --- |
| 编码开始后仍然文档优先 | 文档优先从“编码前准入”变成“变更控制机制” |
| S1 只能按已签收范围施工 | 签收只允许 S1 工程骨架，不自动允许 S2-S7 |
| 任何范围变化先改文档 | 新增能力、依赖、模块、部署方式、AI 边界都必须先修订上游文档 |
| 小实现只能在授权后跟随任务文档 | 已签收并收到对应阶段启动指令后，任务分解内的文件创建、配置占位和验证脚本才可以直接实现 |
| 质量门禁失败不能推进阶段 | 未验证或验证失败不得宣称阶段完成 |
| 质量门禁不能替代变更分级 | 构建通过只能证明代码可运行，不能把 C2-C4 越界请求变成 C0 |

一句话：

> 编码不是文档优先的终点，而是文档开始约束真实变更的时刻。

## 3. 变更分类

编码开始后，所有变更按以下级别处理。

| 级别 | 类型 | 示例 | 处理方式 |
| --- | --- | --- | --- |
| C0 | 已授权小实现 | 已签收并收到阶段启动指令后，按 S1 创建允许的模块、配置 example、脚本骨架 | 可直接实现，完成后记录验证 |
| C1 | 文档同步 | README、任务分解、验证命令文字修正，交接包文案补充但不改变能力边界 | 可直接改文档，必要时同步索引 |
| C2 | 范围边界变化 | S1 想加入登录、菜单、模型调用；AI 工作台新增执行能力；交接包改变任务边界 | 停止编码，先修订路线图、任务分解和冻结清单 |
| C3 | 技术决策变化 | 新增 MQ、ES、Docker 强依赖、换 pnpm；把 P1/P2 AI 能力提前为 P0 | 停止编码，先新增或修订 ADR |
| C4 | 安全/生产高风险 | 生产在线改代码、自动删数据、绕过权限、生产启用开发型 AI | 默认拒绝，除非重新设计并完成安全审计 |

## 4. 可直接实现的变更

以下变更只有在对应阶段已签收、并且维护者明确发出该阶段启动指令后，才可以按任务文档直接实现。当前仍为文档优先阶段时，C0 不成立。

| 阶段 | 可直接实现 | 前提 |
| --- | --- | --- |
| S1 | 工程目录、Maven 多模块、Vue 骨架、PowerShell 脚本骨架、配置 example | `docs/coding-start-signoff.md` 已签收并收到“开始 S1 工程骨架编码” |
| S2 | 登录、用户、角色、菜单、字典、日志 | S1 已验收，且 S2 任务被维护者明确启动 |
| S3 | 模型配置、连接测试、用量记录 | S2 已验收，且 S3 任务被维护者明确启动 |
| S4 | 单表 CRUD 元模型、代码生成、客户拜访记录演示模块 | S3 已验收，且 S4 任务被维护者明确启动 |
| S5 | Windows 开发包 doctor/dev-start/dev-stop/setup-model | S4/S5 依赖关系已确认，且 S5 任务被维护者明确启动 |
| S6 | 生产打包、安装、状态、备份、恢复 | S5 已验收，且 S6 任务被维护者明确启动 |
| S7 | 端到端演示验收 | S1-S6 必要能力已完成，且 S7 验收被维护者明确启动 |

未明确启动的阶段，不得因为“已经开始编码”而提前实现。

## 6. 外部 AI Coding 工具执行规则

外部 AI Coding 工具收到编码任务时，必须先判断变更级别。

| 步骤 | 要求 |
| --- | --- |
| 1 | 读取 `docs/README.md`、当前阶段任务分解和本文 |
| 2 | 先检查签收状态和阶段启动指令，再判断任务属于 C0-C4 哪一级 |
| 3 | 如果是 C2-C4，先停止编码并说明应修订哪些文档 |
| 4 | 如果是 C0-C1，按任务文档小步修改；如果当前仍未签收，C0 自动降级为“只允许修订文档” |
| 5 | 修改后执行最小相关验证 |
| 6 | 输出变更文件、验证结果、风险和是否触发文档同步 |

执行顺序必须固定为“授权判断 -> 变更分级 -> 修改 -> 验证 -> 摘要”。不得先修改或先跑通验证，再回头把越界内容解释为已授权。

| 误用 | 正确处理 |
| --- | --- |
| 代码已经写完且测试通过，所以接受跨阶段功能 | 仍按 C2 处理，暂停合入并回到路线图、任务分解或冻结清单 |
| 新依赖能让构建通过，所以直接保留 | 仍按 C3 处理，先补 ADR 或产品约束 |
| 生产脚本验证通过，所以允许生产执行交接包 | 仍按 C4 处理，默认拒绝并回到安全治理 |
| AI 已经生成补丁，所以可以绕过签收 | 拒绝，签收状态和阶段启动指令优先于生成结果 |

禁止行为：

| 禁止 | 原因 |
| --- | --- |
| 把一次签收解释成所有阶段都可做 | 签收只授权指定阶段 |
| 为了让代码跑通私自引入新依赖 | 破坏技术栈冻结 |
| 在 S1 阶段补登录、菜单或模型调用 | 跨阶段偷跑 |
| 把交接包格式扩展成服务端任意执行能力 | 违反 P0 不自研完整 Agent 和服务端任意 shell 禁止项 |
| 把 AI 使用路径变更当作普通文案修改 | 可能绕过签收包、读者测试和准入审计 |
| 未验证就标记完成 | 违反质量门禁 |
| 发现文档冲突后按代码偏好处理 | 应先修订上游文档或 ADR |

## 7. 变更请求处理模板

当用户或实施人员提出新变更时，按以下格式判断。

| 字段 | 填写要求 |
| --- | --- |
| 请求摘要 | 用户想做什么 |
| 当前阶段 | S1-S7 或文档优先阶段 |
| 变更级别 | C0/C1/C2/C3/C4 |
| 是否可直接编码 | 是/否 |
| 需要先修订的文档 | 列出文档路径 |
| 验证命令 | 列出最小验证命令 |
| 风险 | 数据、安全、依赖、部署、范围风险 |

示例：

| 字段 | 示例 |
| --- | --- |
| 请求摘要 | S1 顺便实现登录页和用户表 |
| 当前阶段 | S1 |
| 变更级别 | C2 |
| 是否可直接编码 | 否 |
| 需要先修订的文档 | `docs/s1-task-breakdown.md`、`docs/basic-admin-spec.md`、`docs/coding-freeze-checklist.md` |
| 验证命令 | 暂不执行编码验证 |
| 风险 | S1 范围扩大，可能导致基础后台边界失控 |

AI 使用路径变更示例：

| 字段 | 示例 |
| --- | --- |
| 请求摘要 | 把外部 AI 交接包改为可直接在生产服务器执行补丁和 shell |
| 当前阶段 | 任意阶段 |
| 变更级别 | C4 |
| 是否可直接编码 | 否 |
| 需要先修订的文档 | `docs/adr/0003-ai-tool-usage-boundary.md`、`docs/security-governance.md`、`docs/release-package-design.md`、`docs/coding-freeze-checklist.md` |
| 验证命令 | 暂不执行编码验证 |
| 风险 | 生产开发型 AI、任意命令执行、源码和数据库结构在线修改风险 |

## 8. 阶段推进规则

进入下一个阶段前，必须满足：

| 条件 | 说明 |
| --- | --- |
| 当前阶段交付物完成 | 对照阶段任务分解 |
| 当前阶段质量门禁已执行 | 通过或明确记录失败和补验 |
| 阶段越界检查通过 | 没有提前实现后续阶段能力 |
| 文档同步完成 | README、准入审计、冻结清单和任务文档无冲突 |
| 用户明确启动下一阶段 | 不能自动从 S1 滑到 S2 |

阶段完成不是一句“已完成”，必须形成阶段关闭证据包。证据包用于证明当前阶段真的完成、没有越界，并且不会自动授权下一阶段。

| 证据项 | 必填内容 | 不合格表现 |
| --- | --- | --- |
| 阶段标识 | 阶段编号、阶段名称、启动依据、启动时间或协作记录 | 只写“本轮完成” |
| 交付物清单 | 对照阶段任务分解列出已创建或修改的文件、脚本、配置、页面、接口或迁移 | 只列部分核心文件，遗漏脚本、配置或文档 |
| 验证结果 | 执行命令、状态、失败原因、日志路径或未执行原因 | 只说“测试通过”但没有命令 |
| 越界检查 | 明确说明是否提前实现后续阶段能力、是否新增冻结外依赖、是否触碰生产禁区 | 没有检查跨阶段内容 |
| 文档同步 | README、阶段任务、质量门禁、变更控制或签收材料是否需要更新 | 实现已变但文档未同步 |
| 残余风险 | 未完成项、暂缓项、环境限制、需要人工复核的事项 | 把 skipped 写成 passed |
| 下一阶段请求 | 如果要进入下一阶段，只能提出请求，不能当作已授权 | 在 S1 总结里直接开始 S2 |

阶段关闭证据包通过后，只表示“当前阶段可以申请关闭”。下一阶段仍必须由维护者明确启动；没有启动指令时，后续请求按 C1 文档同步或 C2-C4 处理，不能默认为 C0。

## 10. 一句话总结

Vibe Boot 的编码后变更控制规则是：已签收的阶段小步实现，未签收的阶段不能偷跑；已冻结的范围不能顺手扩大，所有超出文档的变化先回到文档和 ADR。

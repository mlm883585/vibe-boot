# ADR-0003：AI 工具使用边界

## 状态

已接受。

## 背景

Vibe Boot 的产品定位依赖 AI coding，但“如何使用 AI 工具”如果不先确定，后续实现会出现三类偏差：

| 偏差 | 风险 |
| --- | --- |
| 做成普通 Chat 页面 | 只能问答，不能稳定生成可交付业务模块 |
| 做成完整 AI IDE | 文件、终端、Git、权限隔离复杂度过高，拖慢 MVP |
| 生产环境开放 coding agent | 可能误改源码、执行危险脚本、破坏数据结构 |

因此需要在编码前明确：哪些 AI 能力属于外部开发工具，哪些属于 Vibe Boot 内置工作台，哪些只能在生产业务场景中使用。

## 决策

Vibe Boot MVP 采用“三层 AI 工具使用模式”。

本 ADR 是“如何使用 AI 工具”的上游决策记录。当前口径已分层定稿，不再作为未定问题悬空；但进入实现仍必须完成 `docs/coding-start-signoff.md` 人工签收。后续如要把首版改成完整 AI IDE、生产 Agent 或普通 Chat 外壳，必须先修订本文，并同步冻结清单、需求追踪矩阵、签收包和签收记录。

| 层级 | 工具形态 | 使用者 | 决策 |
| --- | --- | --- | --- |
| 外部 AI Coding 工具 | Codex、Cursor、Claude Code、GitHub Copilot、通义灵码等 | 平台开发者、实施人员 | 负责真实源码修改、运行验证命令、补测试、修 Bug |
| 平台内 AI 工作台 | Vibe Boot 管理端内置页面 | 企业用户、实施人员 | 负责需求澄清、计划、风险、元模型、受控生成、验证摘要 |
| 生产业务 AI | 业务页面中的 AI 功能 | 最终业务用户 | 只做问答、摘要、分类、文案、分析，不做代码修改 |

MVP 不自研完整 AI IDE，也不在服务端直接提供任意 shell 执行能力。

## 用户操作模型冻结

首版必须把“用户如何使用 AI 工具”作为产品约束，而不是实施人员临场发挥。

| 场景 | 用户入口 | AI 承担 | Vibe Boot 承担 | 冻结结论 |
| --- | --- | --- | --- | --- |
| 开发 Vibe Boot 自身 | Codex、Cursor、Claude Code、通义灵码等外部 AI Coding 工具 | 读文档、改源码、跑命令、补测试、输出摘要 | 提供文档索引、ADR、任务单、质量门禁和脚本 | P0 主路径 |
| 企业用户生成业务模块 | Vibe Boot 管理端 AI 工作台 | 澄清需求、生成计划、标注风险、输出元模型或补丁建议 | 管理模型、skills、规则、审计、确认流和验证摘要 | P0 产品化入口 |
| 实施人员落地客户需求 | 外部 AI Coding 工具 + 平台 AI 工作台组合使用 | 处理真实文件修改和问题修复 | 用工作台沉淀需求、计划、生成记录和验证结果 | P0 推荐交付路径 |
| 生产业务 AI | 已部署业务系统中的 AI 功能 | 摘要、分类、问答、文案、数据分析 | 通过模型网关控制权限、用量、脱敏和审计 | 可选能力 |

冻结规则：

| 规则 | 说明 |
| --- | --- |
| 外部 AI Coding 工具不是临时过渡 | P0 将其作为真实源码修改的主执行环境，并通过文档和任务单约束 |
| 平台内 AI 工作台不是完整 IDE | P0 只产品化需求、计划、风险、元模型、补丁建议、验证摘要 |
| 企业用户不直接面对源码细节 | 企业用户主要在工作台确认需求和计划，源码处理可由实施人员或外部 AI 工具完成 |
| 企业用户不需要先掌握 AI Coding 工具 | 工作台必须把业务输入转成结构化需求、准入卡、交接包和验收信息 |
| 外部 AI 交接包不是执行器 | 交接包只用于开发/实施协作，不得作为生产服务器执行补丁、SQL 或 shell 的入口 |
| 外部 AI 交接包不是授权书 | 交接包不能绕过签收状态、阶段启动口令、允许范围和质量门禁 |
| 生产环境不承接开发型 AI | 生产只允许安全业务 AI，代码修改必须回到开发模式 |
| 所有入口共享同一套上下文 | `docs/README.md`、ADR、产品约束、任务分解和质量门禁是共同入口 |

## 具体约束

| 约束 | 决策 |
| --- | --- |
| AI 修改源码 | 仅开发工作区允许，且必须经过签收、阶段启动、计划、diff、验证、摘要 |
| 平台内执行命令 | P0 不做任意终端执行，验证命令由外部 AI 工具或本地脚本执行 |
| 生产代码修改 | 默认禁止 |
| 生产数据库结构修改 | 默认禁止在线修改，只允许通过受控迁移和升级包 |
| 模型接入 | 统一走模型网关，业务模块不得直接散落模型 SDK |
| 上下文入口 | 外部 AI 工具和平台内 AI 工作台都以 `docs/README.md` 作为默认入口 |
| 生成方式 | P0 以元模型、模板、规则和质量门禁为主，不依赖模型自由发挥 |
| 高风险操作 | L2/L3 必须显式确认，不能用一次“继续”覆盖全部风险 |

## P0 AI 工作台边界

| 能力 | P0 决策 |
| --- | --- |
| 模型配置和连接测试 | 做 |
| 需求输入和澄清 | 做 |
| 生成计划和风险摘要 | 做 |
| 生成单表 CRUD 元模型 | 做 |
| 生成文件清单和变更摘要 | 做 |
| 基础 diff 展示 | 做 |
| 外部 AI 交接包 | 做，但只作为开发/实施交接材料，不作为生产执行脚本 |
| 开发工作区本地受控补丁 | P1 前不内置，P1 也不得扩展为服务端任意文件写入 |
| 完整 Agent Runtime | 不做 |
| 服务端任意 shell | 不做 |
| 生产在线改代码 | 不做 |

## 影响

| 影响域 | 结果 |
| --- | --- |
| 产品设计 | AI 工作台是受控代码生产线，不是自由聊天窗口 |
| 工程实现 | S1-S4 优先建设文档、规则、模板、质量门禁，而不是 IDE Agent |
| 安全治理 | 生产环境默认关闭开发型 AI 能力 |
| 用户体验 | 企业用户通过工作台描述业务，实施人员可用外部 AI 工具处理源码细节 |
| 交付闭环 | S7 演示必须证明 AI 工具路径能从需求走到生产安装包 |

## 替代方案

| 方案 | 放弃原因 |
| --- | --- |
| 全部依赖外部 AI Coding 工具 | 企业用户入口不产品化，平台价值不清晰 |
| 首版自研完整 AI IDE | 实现复杂、安全风险高、偏离 Windows Java Admin MVP |
| 生产环境开放代码生成 | 不符合中小企业生产安全和可运维要求 |
| 只做传统代码生成器 | 不能体现 AI 原生需求澄清、计划和演进能力 |

## 一句话总结

Vibe Boot 的 AI 工具策略是：外部 AI Coding 工具负责真实开发，平台内 AI 工作台负责把高频业务生成流程产品化，生产环境只保留安全业务 AI；三者边界清楚，系统才不会在 MVP 阶段失控。

# Vibe Boot AI 工作台基础任务分解

## 1. 文档目的

本文把 `docs/ai-workbench-design.md` 拆解为可执行的 AI 工作台基础任务，明确 P0 做什么、不做什么、按什么顺序做、交付什么以及如何验收。

AI 工作台是 Vibe Boot 的核心入口，但 P0 不能做成完整 IDE、Agent Runtime 或低代码设计器。首版只做“需求输入、上下文摘要、模型调用、计划生成、风险标注、任务审计、结果摘要”的轻闭环。

## 2. P0 总目标

| 目标 | 验收口径 |
| --- | --- |
| 用户可创建 AI 任务 | 在管理端输入自然语言需求并生成任务记录 |
| 模型可调用 | 通过 S3 模型网关完成需求澄清或计划生成 |
| 上下文可追溯 | 展示本次使用的文档、规则和假设 |
| 计划可审查 | 输出变更计划、影响范围、风险等级 |
| 交接包可复制 | 计划可导出为外部 AI Coding 工具交接包 |
| 风险可确认 | L2/L3 风险必须等待用户确认 |
| 过程可审计 | 记录任务、输入、模型、用量、风险、状态 |
| 生产可限制 | 生产默认禁用代码级任务和脚本能力 |

## 3. 前置条件

| 前置项 | 要求 |
| --- | --- |
| S2 基础后台 | 登录、权限、菜单、操作日志可用 |
| S3 模型网关 | 模型配置、连接测试、调用服务、用量记录可用 |
| 文档入口 | `docs/README.md`、产品约束、AI 工具策略、质量门禁已存在 |
| 安全策略 | 生产禁用开发型 AI 能力、密钥脱敏已确认 |
| 数据策略 | 模型调用前的数据分类、最小化、脱敏和出境提示已确认 |
| 前端基础 | 管理端布局、动态菜单、权限按钮可用 |

## 4. 任务顺序

| 顺序 | 任务 | 主要产物 | 依赖 |
| --- | --- | --- | --- |
| 1 | AI 任务表与状态机 | `ai_task`、`ai_conversation`、`ai_message`、状态枚举 | S3 数据库基础 |
| 2 | 工作台菜单与权限 | AI 工作台菜单、任务权限 | S2 菜单权限 |
| 3 | 任务创建 API | 创建任务、获取任务、分页查询、取消任务 | 任务表 |
| 4 | 上下文摘要构建 | 文档引用、规则引用、假设列表 | 文档体系 |
| 5 | 需求澄清调用 | 调用模型网关生成问题或假设 | S3 模型网关 |
| 6 | 变更计划生成 | 输出文件、接口、表、权限、页面、验证计划 | 上下文摘要 |
| 7 | 外部 AI 交接包生成 | 阶段、目标、范围、禁止事项、风险、验证命令、输出格式 | 变更计划 |
| 8 | 风险分级与确认 | L0-L3 风险、确认记录 | 规则文档 |
| 9 | 前端工作台页面 | 输入区、上下文区、计划区、风险区、交接包、角色视图、历史区 | API |
| 10 | 审计与用量关联 | 任务、模型用量、操作日志关联 | S2/S3 |
| 11 | 生产限制与验收 | dev-tools 开关、质量门禁 | 安全治理 |

## 5. 任务明细

### 5.1 AI 任务表与状态机

| 表 | P0 用途 |
| --- | --- |
| `ai_task` | 任务主表，记录任务类型、状态、风险、模型、创建人 |
| `ai_conversation` | 任务会话，可按任务聚合 |
| `ai_message` | 用户输入、AI 输出、系统提示摘要 |
| `ai_context_ref` | 使用过的文档、规则、代码引用 |

P0 任务状态：

| 状态 | 含义 |
| --- | --- |
| draft | 已创建，尚未调用模型 |
| clarifying | 正在澄清需求 |
| planned | 已生成计划 |
| waiting_confirm | 等待用户确认风险或计划 |
| handoff_ready | 已具备可复制外部 AI 交接包 |
| executing_external | 外部 AI Coding 工具或开发工作区受控执行器处理中 |
| verifying | 正在记录或等待验证结果 |
| completed | 已完成 P0 计划/摘要流程 |
| failed | 调用或处理失败 |
| blocked | 缺少模型、环境、签收、需求或验证条件 |
| cancelled | 用户取消 |

P0 不要求任务真正应用代码补丁，代码修改可由外部 AI Coding 工具或后续 S4/S5 能力承接。若出现 `executing_external`，只能表示外部工具或开发工作区受控执行器正在处理交接包，不能表示平台服务端正在执行任意 shell 或写源码。

状态流转约束：

| 约束 | 说明 |
| --- | --- |
| `draft` 不得直接进入 `executing_external` | 必须经过澄清、计划、风险确认和交接包生成 |
| `waiting_confirm` 不得自动完成 | L2/L3 风险必须有确认记录 |
| `handoff_ready` 不等于编码授权 | 未签收或未启动阶段时，只能作为文档和实施交接材料 |
| `completed` 必须有证据 | 至少包含计划、交接包或验证摘要；缺少证据只能为 `failed`、`blocked` 或 `waiting_confirm` |
| 生产模式不得进入 `executing_external` | 生产只允许只读查看或业务 AI 能力 |

`ai_task` 必须能保存 AI 使用准入卡快照。字段可命名为 `admission_card_json`，API 返回统一映射为 `admissionCard`。准入卡只记录编码许可、任务阶段、执行入口、上下文、风险、验证和生产边界结论，不记录模型密钥或用户敏感业务明细。

`ai_task` 或关联表还必须保存 skills 与规则快照。字段可命名为 `skill_snapshot_json`、`rule_snapshot_json` 和 `resolution_trace_json`，API 返回统一映射为 `skillSnapshot`、`ruleSnapshot` 和 `resolutionTrace`。快照用于解释本次任务加载了哪些版本的规则、按什么优先级裁决、为什么阻断或放行。

### 5.2 工作台菜单与权限

| 菜单 | 路由 | 权限 |
| --- | --- | --- |
| AI 工作台 | `/ai/workbench` | `ai:workbench:use` |
| AI 任务记录 | `/ai/tasks` | `ai:task:list` |

P0 可把任务记录列表放在工作台页面内，不必做复杂任务中心。

角色视图必须和 `docs/ai-workbench-design.md` 保持一致。P0 如果暂不做完全独立页面，也必须在任务详情中展示当前角色、可执行动作和禁止动作。

| 角色 | P0 可见重点 | P0 操作边界 |
| --- | --- | --- |
| 企业管理员 | 业务需求、澄清、计划摘要、风险、验收结果 | 可确认需求和风险，不接触源码执行 |
| 实施人员 | 交接包、准入卡、验证命令、风险清单 | 可复制交接包并回填结果，不绕过验证 |
| Java 开发者 | 上下文文件、允许范围、禁止范围、验证命令 | 可在外部 AI Coding 工具中执行源码任务，不改产品决策 |
| 生产用户 | 业务 AI 结果和使用记录 | 不显示交接包、补丁、源码和脚本入口 |

### 5.3 任务创建 API

| API | 要求 |
| --- | --- |
| `POST /api/ai/tasks` | 创建任务，保存用户输入 |
| `GET /api/ai/tasks/page` | 查询历史任务 |
| `GET /api/ai/tasks/{id}` | 查看任务详情、上下文、计划、风险 |
| `POST /api/ai/tasks/{id}/clarify` | 生成澄清问题或假设 |
| `POST /api/ai/tasks/{id}/plan` | 生成变更计划 |
| `POST /api/ai/tasks/{id}/confirm` | 记录用户确认 |
| `POST /api/ai/tasks/{id}/cancel` | 取消任务 |

所有接口必须有权限控制、当前用户记录和中文错误提示。

### 5.4 上下文摘要构建

P0 不做向量检索，采用固定文档入口 + 任务相关引用。

| 上下文 | P0 要求 |
| --- | --- |
| 文档索引 | 默认引用 `docs/README.md` |
| 产品约束 | 引用 `product-constraints.md` |
| AI 工具策略 | 引用 `ai-tooling-strategy.md`、`ai-tool-usage-guide.md` |
| 当前阶段 | 根据任务类型引用阶段任务分解 |
| 质量门禁 | 引用 `quality-gates.md` |
| Skills/规则 | 引用 `skill-rule-design.md`，记录 skill/rule 版本、状态、优先级和来源 |
| 用户输入 | 保存原始需求和确认内容 |
| 用户选择文件 | 只接受 `uploadedFileIds`，逐个鉴权 active 的 txt/md/csv/json，合计不超过 1 MB |

上下文摘要必须说明：

| 内容 | 说明 |
| --- | --- |
| 使用了哪些文档 | 文件名和用途 |
| 当前任务假设 | 用户未明确时的默认假设 |
| 缺少什么信息 | 需要用户补充的问题 |
| 不会做什么 | 根据产品约束排除越界项 |
| 规则快照 | 展示 `skillSnapshot`、`ruleSnapshot`、`blockedRules`、`warnings` 和裁决原因 |
| 数据安全摘要 | 展示数据分类、脱敏策略、是否涉及出境风险和是否需要确认 |

上下文摘要不得只显示“已加载规则”。至少要能让实施人员和开发者看见规则来源、版本、阻断项和警告项。若规则冲突未裁决，任务只能进入 `blocked` 或 `waiting_confirm`。

上下文摘要也不得只写“已脱敏”。涉及业务数据或用户上传内容时，必须说明使用了哪些数据类别、哪些字段被移除或摘要化、是否经过当前用户数据范围过滤、供应商是否存在出境风险。存在 secret 数据时必须进入 `blocked`，不得调用模型。

`uploadedFileIds` 不接受 storage path、文件 URL 或任意服务器路径。代码文件上下文仍由开发工作区和外部 AI Coding 工具按允许范围读取，不通过 `vibe-file` 下载接口绕过开发/生产边界。

### 5.5 需求澄清调用

| 项目 | 要求 |
| --- | --- |
| 输入 | 用户原始需求、上下文摘要、任务类型 |
| 输出 | 澄清问题或明确假设 |
| 模型 | 通过模型网关调用 |
| 记录 | 写入 `ai_message` 和 `ai_usage_log` |
| 失败 | 返回中文原因和可重试建议 |

模型调用失败不得生成空计划或伪造计划。未配置模型、限流、配额耗尽、超时或网络失败时，任务状态必须进入 `blocked` 或 `failed`，并保存错误码、中文原因和 traceId。

澄清问题应聚焦字段、权限、数据范围、页面、验证和风险，不做泛泛聊天。

### 5.6 变更计划生成

P0 的计划生成是 AI 工作台的核心交付。

| 计划项 | 要求 |
| --- | --- |
| 功能摘要 | 用户想要什么 |
| 涉及阶段 | 属于 S1-S7 哪个阶段 |
| 文件影响 | 后端、前端、SQL、文档、脚本可能影响 |
| 数据库影响 | 新表、字段、迁移、初始数据 |
| 权限影响 | 菜单、按钮、接口权限 |
| 前端影响 | 页面、路由、API、组件 |
| 验证计划 | Maven、npm、脚本或人工验收 |
| 不做事项 | 明确排除越界范围 |

P0 计划可以作为外部 AI Coding 工具的施工提示，但不直接在服务端执行文件修改。

### 5.6.1 外部 AI 交接包生成

AI 工作台必须能把用户确认后的计划转换为可复制的外部 AI Coding 工具交接包。

| 字段 | P0 要求 |
| --- | --- |
| 任务标题 | 一句话说明要做什么 |
| 当前阶段 | S1-S7 或后续阶段 |
| 业务目标 | 用户真正想解决的问题 |
| 需求澄清结论 | 字段、权限、菜单、数据范围、边界 |
| 允许修改范围 | 目录、模块、页面、SQL、测试范围 |
| 禁止事项 | 不新增依赖、不改安全核心、不碰生产配置等 |
| 风险等级 | L0-L3，说明是否已确认 |
| AI 使用准入卡 | 编码许可、任务阶段、执行入口、上下文、风险、验证和生产边界 |
| 预期验证命令 | Maven、npm、PowerShell 或人工验收 |
| 相关文档 | 至少包含 `docs/README.md` 和阶段任务文档 |
| 输出格式 | 文件清单、验证结果、风险、后续建议 |

交接包只能用于开发模式或实施人员交接，不得被解释为生产服务器在线执行补丁、shell 或数据库结构修改的授权。

### 5.7 风险分级与确认

| 风险 | 处理 |
| --- | --- |
| L0 文档/解释 | 可直接完成 |
| L1 普通 CRUD 计划 | 计划生成后可进入待执行 |
| L2 改表结构、权限、依赖 | 必须用户确认 |
| L3 删除数据、认证核心、生产发布 | 必须二次确认，建议备份 |

确认记录必须包含用户、时间、确认内容和风险项。

### 5.8 前端工作台页面

| 区域 | P0 要求 |
| --- | --- |
| 任务输入区 | 任务类型、模型、自然语言需求 |
| 上下文区 | 展示使用的文档和规则摘要 |
| 澄清区 | 展示问题、用户回答、假设 |
| 计划区 | 展示变更计划 |
| 风险区 | 显示风险等级和确认按钮 |
| 结果区 | 显示任务状态、模型用量、失败原因 |
| 角色视图区 | 显示当前角色、允许动作和禁止动作 |
| 历史区 | 展示最近任务 |

页面必须是工具界面，不做营销文案和装饰性布局。

### 5.9 审计与用量关联

| 记录 | 要求 |
| --- | --- |
| AI 任务 | 任务状态、类型、风险、创建人 |
| 模型用量 | 关联 taskId |
| 用量摘要 | 记录模型、purpose、token、耗时、错误和配额状态 |
| 操作日志 | 创建任务、确认风险、取消任务 |
| 消息记录 | 用户输入、AI 输出、系统摘要 |
| 上下文引用 | 文件名、规则名、用途 |
| 规则快照 | skill/rule 版本、状态、优先级、checksum 和来源文档 |
| 裁决记录 | 规则冲突、风险等级、阻断项、警告项和确认记录 |
| 数据安全 | dataClasses、redactionPolicy、providerRegionRisk、是否经过数据权限过滤 |

### 5.10 生产限制

| 能力 | 生产默认 |
| --- | --- |
| 需求澄清 | 可选开启 |
| 文档问答 | 可选开启 |
| 生成变更计划 | 可选开启，但标记只读 |
| 代码补丁生成 | 关闭 |
| 应用补丁 | 关闭 |
| 脚本执行 | 关闭 |
| 数据库结构修改 | 关闭 |

生产环境任何开发型能力都必须受 `vibe.ai.dev-tools-enabled=false` 约束。

## 9. 禁止越界项

| 禁止 | 原因 |
| --- | --- |
| 服务端直接执行任意 shell | 安全风险高，P0 不做 |
| 服务端直接改源码 | P0 由外部 AI Coding 工具或后续受控补丁承接 |
| 做完整 diff viewer | P1 优化 |
| 做向量检索知识库 | P1/P2 |
| 做自动测试生成 | P1 |
| 做代码生成模板引擎 | S4 范围 |
| 做生产在线改代码 | 违反安全治理 |
| 做复杂 Agent 工具调用系统 | 偏离 MVP |

## 10. 验证门禁

AI 工作台基础完成后必须满足：

| 门禁 | 验收方式 |
| --- | --- |
| 后端构建 | `mvn -pl vibe-starter -am test` 或快速构建 |
| 前端构建 | `npm run build` |
| 任务创建 | 用户可创建 AI 任务 |
| 模型调用 | 澄清或计划生成通过模型网关完成 |
| 上下文摘要 | 页面展示使用的文档和假设 |
| Skills/规则快照 | 页面展示已加载 skill/rule 版本、来源、优先级、阻断项和警告项 |
| 计划生成 | 输出结构化变更计划 |
| 交接包生成 | 可复制外部 AI 交接包且字段完整 |
| 角色视图 | 企业管理员、实施人员、Java 开发者、生产用户的入口和禁止动作可区分，至少在任务详情中明确 |
| 状态机 | `draft` 到 `completed/failed/blocked` 的关键状态可保存和展示，不允许跳过确认直接执行 |
| 风险确认 | L2/L3 风险进入等待确认 |
| 用量关联 | 模型用量记录关联 taskId |
| 权限控制 | 无权限用户不能使用工作台 |
| 生产限制 | 生产默认关闭开发型任务 |
| 冲突阻断 | 违反 active must/must-not 规则时不得生成可执行交接包 |

## 11. AI 实现提示

外部 AI Coding 工具执行本阶段时必须遵守：

| 规则 | 说明 |
| --- | --- |
| 先读文档 | `docs/README.md`、本文、`ai-workbench-design.md`、`ai-tooling-strategy.md` |
| 先加载规则 | 读取 `skill-rule-design.md` 并形成 `skillSnapshot`、`ruleSnapshot` |
| 不写 Agent Runtime | P0 只做任务和计划闭环 |
| 不绕过模型网关 | 所有模型调用经过 S3 能力 |
| 不暴露密钥 | 上下文、日志、前端响应都不得出现明文 API Key |
| 数据最小化 | 不把整库、完整日志、secret 数据或无关业务数据发送给模型 |
| 风险必须显式 | 计划里必须标注 L0-L3 |
| 失败必须中文化 | 模型失败、权限失败、验证缺失都要中文说明 |

## 12. 完成定义

AI 工作台基础只有在以下条件同时满足时才算完成：

| 条件 | 说明 |
| --- | --- |
| 输入闭环 | 用户可创建任务并提交需求 |
| 模型闭环 | 能通过模型网关生成澄清或计划 |
| 上下文闭环 | 使用了哪些文档、规则和假设可见 |
| 规则闭环 | 使用了哪些 skill/rule 版本、冲突如何裁决、阻断或警告原因可见 |
| 交接闭环 | 用户确认后的计划能转换为外部 AI Coding 工具交接包 |
| 角色闭环 | 企业用户、实施人员、开发者和生产用户的可见信息和操作边界不混用 |
| 状态闭环 | 任务状态能解释当前处于澄清、计划、确认、交接、验证、完成、失败或阻塞哪一步 |
| 风险闭环 | 风险等级和确认记录可追踪 |
| 审计闭环 | 任务、消息、用量、操作日志可关联 |
| 前端闭环 | 工作台页面可用且状态清晰 |
| 生产边界 | 生产默认不允许开发型任务 |

## 13. 一句话总结

AI 工作台基础阶段只做“把用户需求变成可审查计划”的闭环，不做自动改代码的黑盒。它先把上下文、模型、风险、任务和审计打通，后续 S4 再接代码生成。

# Vibe Boot S7 MVP 演示验收剧本

## 1. 文档目的

本文定义 Vibe Boot MVP 的端到端演示验收剧本。它不是新的业务需求文档，而是把 `docs/customer-visit-demo-spec.md` 中的“客户拜访记录”模块放入完整交付链路中验证：开发包、模型接入、AI 工作台、代码生成、本地预览、生产打包、Windows 安装、备份恢复。

S7 的目标不是展示更多功能，而是证明 Vibe Boot 的核心承诺成立：

> 中国中小企业用户拿到 Windows 开发包后，只需要配置大模型，就能在受控规则下生成真实业务代码，并打包为可安装、可运行、可备份的生产系统。

## 2. 验收总目标

| 目标 | 验收含义 |
| --- | --- |
| 开发环境可获得 | 解压后可通过脚本完成诊断和启动 |
| 模型可接入 | 用户能配置 OpenAI 兼容模型并完成连接测试 |
| AI 工作台可用 | 能创建任务、澄清需求、生成计划、展示风险 |
| AI 使用路径可解释 | 用户能按首次使用引导完成模型配置、任务创建，并获得外部 AI 交接包 |
| 代码生成可控 | 生成 Java、Vue、SQL、菜单权限和说明文档 |
| 本地预览可用 | 客户拜访记录页面能完成核心 CRUD |
| 质量门禁可信 | 后端和前端验证结果真实记录，不假装通过 |
| 生产包可交付 | 能生成 Windows 安装包并在新环境安装启动 |
| 运维动作可执行 | status、backup、restore 至少具备可验证路径 |

## 4. 演示角色

| 角色 | 责任 |
| --- | --- |
| 企业管理员 | 描述业务需求、回答澄清、确认计划和风险、查看验收结果 |
| 实施人员 | 解压开发包、运行脚本、配置模型、发起 AI 生成任务 |
| 管理员 | 登录后台、配置用户角色、确认菜单和权限 |
| Java 开发者 | 必要时使用外部 AI Coding 工具处理源码任务并回填验证结果 |
| 销售人员 | 新增、编辑、查询自己的客户拜访记录 |
| 销售主管 | 查看全部客户拜访记录，验证数据范围说明 |

如果 P0 尚未完整实现数据权限，演示必须明确标注“数据权限已定义，当前能力需等待基础能力接入后完全生效”，不得以管理员账号替代所有角色验收。
演示必须区分企业管理员、实施人员和 Java 开发者：企业管理员不应直接面对源码目录或命令行；实施人员负责交接和回填；Java 开发者只在开发工作区或外部 AI Coding 工具中处理源码。

## 5. 演示环境

| 环境项 | 要求 |
| --- | --- |
| 开发机器 | Windows 10/11 或 Windows Server 2019+ |
| 生产测试环境 | 另一台 Windows 机器，或同机干净目录模拟 |
| JDK | 使用开发包或生产包内置 JDK 17 runtime |
| Maven | 开发包内置 Maven 3.8.x，并默认配置国内镜像 |
| Node.js | 开发包内置 Node.js 20.19+ LTS，并默认配置国内 npm 镜像 |
| 网络模式 | 至少演示 online 或 mirror 模式；若声明内网包能力，必须展示缓存清单和缺失依赖诊断 |
| MySQL | MySQL 8，连接信息写入本地 ignored 配置 |
| Redis | Redis 7 或兼容版本，生产可连接外部实例 |
| 模型服务 | OpenAI 兼容接口，API Key 不进入 Git 和生产默认包 |
| 日志目录 | 脚本、后端、前端、安装日志统一落入 `logs/` |

## 6. 演示脚本

| 顺序 | 步骤 | 输入 | 预期输出 | 证据 |
| --- | --- | --- | --- | --- |
| 1 | 准备开发包 | 解压 Vibe Boot 开发包 | 目录结构完整，包含 `runtime/`、`scripts/`、`docs/` | 解压目录截图或目录清单 |
| 2 | 环境诊断 | 执行 `scripts/doctor.ps1` | 输出 JDK、Maven、Node、MySQL、Redis、端口、镜像检查结果 | `logs/scripts/doctor-*.log` |
| 3 | 启动开发模式 | 执行 `scripts/dev-start.ps1` | 后端、前端和必要开发服务启动 | 启动日志和访问地址 |
| 4 | 配置模型 | 执行 `scripts/setup-model.ps1` 或进入模型配置页面 | 模型连接测试成功，密钥脱敏展示 | 模型配置记录和连接测试日志 |
| 5 | 登录后台 | 使用管理员账号登录 | 进入管理端首页，菜单可见 | 登录截图或操作日志 |
| 6 | 打开 AI 工作台 | 新建 AI 任务 | 任务进入 `draft` 或 `planned` 状态 | AI 任务编号 |
| 7 | 输入需求 | 输入客户拜访记录原始需求 | AI 提出澄清问题 | 澄清记录 |
| 8 | 确认澄清 | 接受 `docs/customer-visit-demo-spec.md` 默认答案 | AI 生成字段、权限、页面、SQL 计划 | 生成计划 |
| 9 | 检查角色视图 | 分别以企业管理员、实施人员或开发者视角查看任务 | 可见信息和允许动作不同，企业管理员不显示源码执行入口 | 角色视图截图或任务详情 |
| 10 | 生成外部 AI 交接包 | 从工作台导出或复制交接包 | 包含阶段、目标、范围、禁止事项、风险、验证命令和输出格式 | 交接包文本或截图 |
| 11 | 风险确认 | 用户确认计划和风险 | 高风险项明确，无临时绕过质量门禁，任务状态进入 `waiting_confirm` 或确认后进入 `handoff_ready` | 风险摘要和状态记录 |
| 12 | 生成模块 | 应用客户拜访记录生成任务 | 生成后端、前端、SQL、菜单权限和说明文档 | 生成文件清单 |
| 13 | 运行验证 | 执行后端验证和前端构建 | 验证结果 passed；失败时任务不得标记 completed | 验证结果 JSON 或日志 |
| 14 | 本地预览 | 访问客户拜访记录页面 | 可新增、查询、编辑、删除记录 | 页面截图和操作日志 |
| 15 | 生产打包 | 执行 `scripts/build-prod.ps1` | 生成 Windows 生产安装包 | 安装包文件名和构建日志 |
| 16 | 生产安装 | 在生产测试环境以 lan 模式、测试专用可信 PKCS12 执行 `scripts/install.ps1`；另验证缺少 TLS 时被阻断 | HTTPS 服务安装、迁移和回环管理端口健康检查成功；LAN HTTP 安装失败 | 安装日志与 TLS 预检结果 |
| 17 | 状态检查 | 执行 `scripts/status.ps1`，并验证管理端口非回环不可达和一个依赖故障场景 | 脚本通过 `127.0.0.1:8081` 检查；健康时退出码 0；依赖故障时 liveness 仍 UP、readiness 非 UP 且退出码 21 | status 输出和健康响应证据 |
| 18 | 生产验证 | 通过 HTTPS 登录生产环境并访问客户拜访记录 | 页面可用，Cookie 带 Secure/HttpOnly/SameSite=Strict，写请求 CSRF 生效，数据可写入生产库且无开发型 AI 入口 | 生产页面、响应头或日志证据 |
| 19 | 备份 | 停止测试服务后执行 `scripts/backup.ps1` | 生成包含数据库、文件、配置、版本和脱敏 manifest 的敏感备份，ACL 与日志符合约束 | 备份包路径和 ACL 检查 |
| 20 | 恢复与升级回滚演练 | 在隔离测试环境执行 `scripts/restore.ps1`，并模拟升级迁移或健康检查失败 | 同版本恢复成功、跨版本恢复被阻断；失败升级保持服务停止，并可用同一次回滚点整套恢复 | 恢复日志、失败状态和健康检查 |

## 7. 核心业务验收

| 业务动作 | 验收标准 |
| --- | --- |
| 新增拜访记录 | 必填校验生效，客户名称、拜访时间、拜访方式、纪要可保存 |
| 列表查询 | 新增记录出现在列表，可按客户名称、拜访时间、状态筛选 |
| 编辑记录 | 可修改状态、纪要和下一步动作 |
| 删除记录 | 使用逻辑删除，删除后列表不再展示 |
| 菜单权限 | 菜单和按钮权限标识与 `docs/customer-visit-demo-spec.md` 一致 |
| 接口权限 | 未授权访问应返回统一错误码，不允许匿名写入 |
| 数据权限 | 销售人员和主管的数据范围按当前实现能力展示或明确标注限制 |

## 8. 质量验收矩阵

| 验收域 | 必须证明 | 失败时结论 |
| --- | --- | --- |
| Windows 开发包 | doctor、dev-start、dev-stop 可运行 | 不能宣称开发包可交付 |
| 国内镜像 | Maven/npm 默认使用国内镜像或可配置 | 不能宣称面向中国用户低门槛 |
| 弱网/内网诊断 | doctor 能说明网络模式、镜像可达性、runtime manifest 和缓存缺失 | 不能宣称低环境门槛 |
| 模型接入 | OpenAI 兼容模型连接测试成功 | 不能进入 AI 工作台生成演示 |
| AI 工作台 | 任务、澄清、计划、风险、摘要可追踪 | 不能宣称 AI 开发流程可控 |
| AI 使用路径 | 首次使用引导和外部 AI 交接包可展示 | 不能宣称中小企业用户低门槛可使用 |
| 角色和状态 | 企业用户、实施人员、开发者入口可区分，任务状态可追踪 | 不能宣称平台适合非技术企业用户 |
| 代码生成 | Java、Vue、SQL、权限菜单均生成 | 不能宣称超过传统低代码 |
| 验证门禁 | Maven 和 npm 验证结果真实记录 | 不能把任务标记为完成 |
| 本地预览 | 客户拜访记录 CRUD 可用 | 不能进入生产打包验收 |
| 生产安装 | build-prod、install、status 可用 | 不能宣称生产可交付 |
| 认证与传输 | PBKDF2 参数、登录限流、HttpOnly Cookie、CSRF、LAN HTTPS 和回环管理端口可验证 | 不能宣称生产认证安全边界成立 |
| 发布通道 | 生产环境只接收 build-prod 产物、install/upgrade 和迁移流程 | 不能宣称开发到生产闭环成立 |
| 文件基础 | 文件管理页可验证白名单上传、鉴权下载、图片预览、危险类型拒绝、内部路径隐藏和删除失败状态 | 不能宣称 P0 基础服务完整 |
| 备份恢复 | 停服备份、敏感资产保护、同版本恢复和升级失败整套回滚有可验证路径 | 不能宣称可运维 |

## 9. 不通过条件

| 不通过情况 | 原因 |
| --- | --- |
| 只能手工改源码才能演示 | 未验证 AI 工作台和生成闭环 |
| 不能导出或展示外部 AI 交接包 | 未验证平台把业务需求交给外部 AI Coding 工具的产品化能力 |
| 企业管理员必须打开源码目录或命令行才能完成任务 | 未验证非技术企业用户路径 |
| 任务没有状态记录或失败后仍显示 completed | 未验证 AI 工作台状态机和质量门禁 |
| 只生成前端或只生成后端 | 不是完整业务交付 |
| SQL 没有版本化迁移 | 破坏升级和生产安装 |
| 权限标识缺失 | 违反后台系统安全底线 |
| 验证命令失败却显示通过 | 破坏质量门禁可信度 |
| 生产包包含 API Key 或本地密码 | 违反密钥安全 |
| 生产 LAN 使用 HTTP、Token 进入 Web Storage、Cookie 写请求无 CSRF 或 Actuator 管理端口对外监听 | 违反认证与网络暴露底线 |
| 上传接口接受路径穿越、伪装类型、超限文件，或响应暴露存储路径 | 任意文件写入或信息泄漏风险 |
| 文件上传后自动进入模型上下文，或文档宣称已杀毒 | 违反数据最小化和能力真实性 |
| 生产模式仍可在线改代码 | 违反生产安全边界 |
| 复制开发源码或开发数据库到生产环境 | 绕过受控发布通道 |
| 在生产执行外部 AI 交接包、补丁、临时 SQL 或 shell | 把开发/实施材料误用为生产发布 |
| 安装失败后留下半成品服务且无清理提示 | 生产交付不可接受 |
| 备份包缺少版本和配置摘要 | 后续恢复不可追踪 |
| 备份 manifest 或日志包含配置值、密码、Token、API Key 或业务数据摘要 | 敏感信息泄漏 |
| 跨版本恢复仅提示后继续，或迁移失败后只换回旧 jar | 程序、schema 和数据状态可能不一致 |

## 11. S7 禁止越界

| 禁止项 | 说明 |
| --- | --- |
| 临时新增业务模块 | S7 只验收客户拜访记录，不扩展 CRM/进销存 |
| 临时引入新中间件 | 不为演示引入 MQ、搜索、报表引擎等 |
| 临时绕过质量门禁 | 构建失败不能通过手工截图掩盖 |
| 手工改生产库伪造结果 | 所有结构变更必须来自迁移或安装流程 |
| 跳过 build-prod/install/upgrade 发布链路 | 无法证明生产包可交付 |
| 用管理员账号替代权限验收 | 至少要说明角色差异和当前限制 |
| 为演示打开生产代码编辑 | 生产模式必须禁用开发型 AI 能力 |

## 12. 完成定义

S7 完成必须同时满足：

| 条件 | 标准 |
| --- | --- |
| 全流程可重复 | 另一位实施人员按本文能复现主要流程 |
| 证据可追踪 | 每个关键步骤都有日志、截图、任务记录或命令输出 |
| 失败可解释 | 任一失败都有中文原因、日志路径和下一步建议 |
| 边界不扩散 | 演示没有突破 P0/P1/P2 范围约束 |
| 生产安全成立 | 生产包不包含开发型代码修改能力和敏感密钥 |
| 发布通道成立 | 开发成果通过 build-prod、install/upgrade、Flyway 迁移和健康检查进入生产，没有复制源码、交接包执行或手工 SQL 旁路 |
| 使用路径成立 | 首次使用引导、企业用户确认路径和外部 AI 交接包都有证据 |
| 角色状态成立 | 企业用户不接触源码执行，实施人员和开发者接力清晰，任务状态能解释每一步为什么可继续或必须停止 |

## 13. 一句话总结

S7 不是“演示一个页面”，而是验证 Vibe Boot 的整条产品链路：Windows 拿来能跑，AI 能受控生成真实代码，质量门禁能证明可用，生产包能安装运维。
